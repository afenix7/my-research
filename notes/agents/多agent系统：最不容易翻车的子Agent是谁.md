# 多agent系统：最不容易翻车的子Agent是谁？

> 本文是《Building multi-agent systems: when and how to use them》系列第③期笔记。
> 前两期分别讨论了：何时采用多agent架构（第①期）、agent边界应该如何拆分（第②期）。

## 概述

本视频回答了一个具体的实操问题：在决定引入多agent架构之后，最安全、最不容易翻车的第一刀应该切在哪里？答案出乎意料地"平凡"：**验证子Agent（Validation Agent）**，而不是那些看起来更聪明的推理Agent或写作Agent。视频深刻辨析了"编写测试"与"执行验证"的本质区别（白盒任务 vs. 黑盒任务），给出了防止验证Agent偷懒的四条铁律，并以系列落地检查清单作为收尾。

---

## 核心内容

### 一、验证Agent：最稳的拆分起点

Anthropic（Cloud/Claude）在其多agent系统指南中，没有推荐那些花哨的写作模式，而是着重强调了一个看起来"不那么聪明"的角色——**验证子Agent（Validator Agent）**。

#### 典型架构模式

在这种模式中存在两个角色：

| 角色 | 职责 | 上下文需求 |
|------|------|-----------|
| **主Agent** | 真正干活：写代码、生成内容 | 需要完整上下文，必须理解"为什么" |
| **验证Agent** | 只回答一个问题：结果是否满足预先定义的标准 | 不需要任何过程上下文 |

验证Agent完全不关心过程——它不需要知道代码为什么这么写，不需要理解业务逻辑，只需要**拿到产物、对照标准、机械地执行检查**。

---

### 二、关键辨析：编写测试 vs. 执行验证

视频着重解答了一个容易混淆的矛盾：

- 第②期说"把测试拆分出去是错误的传声筒架构"
- 第③期又说"拆分验证是最安全的"

**这两者有本质区别：**

#### 编写测试 = 白盒任务

编写测试必须深度理解代码内部：
- 必须理解业务逻辑
- 必须理解代码结构
- 必须理解开发者为什么这么实现
- 需要**大量上下文**

如果强行把"编写测试"拆分给一个独立Agent，它就会变成瞎子，必须不断向主Agent询问，从而掉进**传声筒陷阱**（Telephone Anti-pattern）——信息在Agent之间反复传递，产生损耗和扭曲。

#### 执行验证 = 黑盒任务

执行验证只需要最少的上下文：
- 拿到产物
- 对照标准
- 机械执行检查
- 不需要知道代码是怎么写的，也不需要知道"为什么用for循环"

#### 生活类比

| 角色 | 对应任务 | 特征 |
|------|---------|------|
| 学生写作文 | 编写测试 | 必须自己构思，这叫"上下文内句执行" |
| 老师判卷子 | 执行验证 | 不关心写作时的心理活动，只看字数够不够、有没有跑题 |

**核心结论：** 验证任务天然符合"以上下文为中心的分解原则"——它是上下文需求最干净的一类工作。

---

### 三、验证Agent的广泛应用场景

理解了这个逻辑之后，验证Agent的适用范围远超代码测试：

| 场景 | 主Agent职责 | 验证Agent职责 |
|------|------------|--------------|
| **合规检查** | 写合同 | 拿着公司法务手册核对条款，不用懂法律逻辑，只看条款在不在 |
| **事实核查** | 写文章 | 专门去搜 Google，验证每一个数据的真实性 |
| **格式校验** | 生成数据 | 专门检查输出的 JSON 字段是否正确 |
| **代码测试** | 写代码 | 运行测试套件，报告所有失败项 |

一句话定位：**验证Agent是系统中成本最低但收益最高的守门员。**

---

### 四、实战陷阱：过早胜利（Premature Victory）

即使是这么安全的拆分起点，实战中也存在一个巨大陷阱——**验证子Agent的偷懒倾向**。

**根本原因：**
- LLM 天生倾向于顺从用户
- LLM 倾向于尽快结束对话

**典型表现：**
- 验证者只跑了一两个最简单的测试
- 发现通过就立刻宣布"大功告成"
- 但这是假象——一旦上线遇到复杂边界情况，系统就会崩溃

---

### 五、防偷懒四条铁律（建议写入 system prompt）

Anthropic 官方给出了四条防止验证Agent偷懒的具体规则：

#### 铁律一：使用具体标准

- **错误写法：** "确保能运行"
- **正确写法：** "运行完整的测试套件，并报告所有的失败项"

#### 铁律二：强制全面检查

- 明确要求覆盖边界情况
- 示例边界场景：空值、超长文本、特殊字符

#### 铁律三：负向测试（Negative Testing）

- 指令验证Agent输入一些**会导致失败的数据**
- 确认系统真的报错了
- 判断依据：如果无论输入什么它都说"成功"，则测试是假的

#### 铁律四：使用强祈使句

- **示例指令：** "你**必须**运行完所有测试，才能标记为通过"
- 使用强制性动词，不留模糊空间

---

### 六、系列总结：多Agent设计落地检查清单

在动手写代码前，问自己三个问题：

**问题一：真的有物理限制吗？**
- 单体架构能解决的，坚持用单体
- 不要为了"多Agent显得高级"而引入不必要的复杂性

**问题二：是按上下文拆分的吗？**
- 拆分边界应基于**知识需求**（哪些Agent需要哪些上下文）
- 而不是盲目模仿流水线工厂的分工方式

**问题三：有清晰的验证点吗？**
- 能否定义出明确的验证标准
- 让子Agent进行真正的黑盒验证（而非假装验证）

#### 系列一句话总结

> 从单体开始，只在真实约束出现时引入多Agent；多Agent最稳的拆分起点是**验证子Agent**，而不是更复杂的思考Agent。

## 关键术语表

| 术语 | 解释 |
|------|------|
| **验证子Agent（Validation Agent）** | 只负责对照预定义标准检查产物是否合格的子Agent，不需要任何过程上下文 |
| **白盒任务** | 需要深度了解内部实现逻辑才能完成的任务，如编写测试代码 |
| **黑盒任务** | 只需要关注输入输出是否符合标准的任务，不关心内部实现，如执行验证 |
| **传声筒陷阱（Telephone Anti-pattern）** | 拆分了需要大量上下文的任务，导致子Agent必须反复向主Agent询问，信息在传递中失真 |
| **以上下文为中心的分解原则** | 按照"每个Agent需要什么上下文"来划分边界，而非按功能类型划分 |
| **过早胜利（Premature Victory）** | 验证Agent只测试了简单用例就宣布成功，实际上遗漏了大量边界场景 |
| **负向测试（Negative Testing）** | 主动输入应该导致失败的数据，验证系统是否真的正确报错 |
| **强祈使句** | 在 system prompt 中使用"必须"等强制性动词，防止Agent找借口偷懒 |
| **单体架构（Monolith）** | 所有逻辑在单个Agent中完成，没有子Agent拆分，适合大多数场景 |
| **物理限制** | 迫使必须引入多Agent的真实约束（如上下文窗口溢出、并行执行需求），而非主观臆断 |

---

## 总结

本视频最核心的贡献是从第一性原理出发，解释了为什么验证Agent是多Agent系统最安全的入口：因为验证任务的上下文需求是零，天然适合隔离成独立Agent。而那些看起来更有趣的"写作Agent"、"推理Agent"往往需要大量上下文，贸然拆分反而会制造传声筒问题。

**关键注意点：**
- 把"编写测试"和"执行验证"混为一谈是常见错误，两者一个是白盒、一个是黑盒
- 验证Agent几乎必然会偷懒，必须在 system prompt 中用四条铁律约束
- 引入多Agent之前，永远先问"是否真有物理限制"——单体解决不了的再拆
