# AI编程最佳工作流GSD完全指南

## 概述

本视频介绍了一个名为 GSD 的开源规范驱动工作流，旨在解决AI编程中"越写越烂"的核心问题。GSD 在 GitHub 上已有约3万 stars，通过30多个命令将项目开发的每个流程切分精细，并通过子代理并行执行、持久化文档链路、阶段性状态追踪三大机制，从根本上解决了上下文臃肿、需求散落、会话重置后需重来等痛点。视频详细介绍了 GSD 的完整工作流（初始化→讨论→计划→执行→验收）、文件体系、适用场景以及缺点。

---

## 核心内容

### 一、问题背景：没有工作流的AI编程会怎样

**典型崩坏路径：**
```
有个想法 → 让AI润色 → 确认没问题 → 开始编码
→ 加功能 → 修问题 → 加约束 → 上下文越来越长
→ AI产生幻觉、走捷径、忘记之前的约束
→ 代码越来越烂 → 项目崩盘 → 推倒重来
```

**四大根源问题：**

| 问题 | 原因 |
|------|------|
| 上下文越来越长 | AI自动压缩上下文时会损失大量有效信息 |
| 需求散落各处 | 加功能、改按钮、加验证等指令散落在对话中，无法形成统一约束 |
| 主线程占用过高 | 没有子代理时，所有工作都在主对话进行，消耗大量上下文 |
| 会话重置即清零 | 关闭 Claude Code 或新开对话后，所有上下文需重新交代 |

### 二、已有工作流的对比

在介绍 GSD 之前，视频回顾了市面上其他工作流：

| 工作流 | 特点 | 缺点 |
|--------|------|------|
| **Super powers** | 写计划→头脑风暴→执行计划 | 无子代理，文档不完整 |
| **Workflows** | 头脑风暴→计划→执行 | 无子代理，文档不完整 |
| **MANU** | 生成计划文件，关闭对话后也能恢复任务 | 无完整的阶段文档体系 |

**共同缺点：**
- 没有用子代理解决上下文臃肿问题
- 没有生成覆盖完整阶段的详细文档体系

### 三、GSD 是什么

**GSD（Get Stuff Done）** 是一个开源的**规范驱动工作流**：
- GitHub stars：约3万
- 核心机制：30多个命令，把项目开发的每个流程切分精细
- 文件体系：每个流程阶段都会产生相关文件，文件之间有总分关联关系
  - **全局文件**：整个项目生命周期都生效
  - **阶段文件**：仅在对应阶段生效
- 并行执行：自动标记哪些计划可并行，开启多个子代理执行

**GSD 如何解决四大问题：**

| 原有问题 | GSD 的解决方式 |
|---------|--------------|
| 上下文过长 | 开启子代理，将需求拆分为多个并行/串行计划，主Agent上下文占用极少 |
| 需求散落 | 生成关联的文档体系，总文件统一管理所有阶段文档 |
| 状态丢失 | `state.md` 文件追踪每个阶段完成百分比，任何地方启动都能读取 |
| 会话重置 | 持久化文件体系，新会话直接读取已有文档继续执行 |

### 四、完整工作流详解

#### 第一步：初始化（new project / map code base）

**场景一：已有代码项目**
1. 先执行 `map code base`
2. GSD 扫描代码库中所有代码和文档
3. 生成：技术栈描述、架构说明、目录结构、项目概况

**场景二：全新项目**
执行 `new project` 后，生成四个**全局文件**（整个项目生命周期中所有阶段都会读取这些文件作为参考）：

| 文件 | 内容 | 作用 |
|------|------|------|
| `project.md` | 项目描述、主要功能、MVP阶段目标 | 核心参考文档 |
| 功能验收清单 | 列出 MVP 阶段的功能列表 | 指导计划生成 |
| `roadmap.md` | 将需求分为多个阶段，每个阶段关联功能 | 阶段规划路线图 |
| `state.md` | 记录每个阶段的完成百分比和运行情况 | 全局状态追踪 |

> `state.md` 是 GSD 实现"会话无关"的关键——无论在哪里启动，都会先读取这个文件了解当前进度。

---

#### 第二步：讨论（discuss）

**命令格式：** `discuss <阶段编号>`（如 `discuss 1`）

**是否必须：**
- 可以是可选步骤
- 如果在 `new project` 时已提供详尽的 PRD 文档，可以直接跳过讨论，直接生成计划

**执行过程：**
- AI 通过 Claude Code 的 `ask` 工具向用户提问，收集需求细节
- 收集完所有问题后，生成 `context.md`（需求确认文件）

**产物：**
- `context.md`：该阶段的需求确认文档，记录讨论结果和用户意图

---

#### 第三步：生成计划（plan）

**作用：** 针对特定阶段生成具体执行计划

**阶段结构：** 每个阶段都是完整的一套流程：
```
讨论（discuss） → 计划（plan） → 执行（execute） → 验收（validate）
```
阶段之间可能独立，也可能相互依赖，GSD 会自动设计。

**执行过程：**
- GSD 调用子代理进行分析和设计
- 生成以 `.plan`（或 PLAN）结尾的具体计划文件

**Wave 编号机制（核心）：**
- 每个 plan 文件包含一个 **wave 编号**
- 相同 wave 编号 → 可以并行执行
- 不同 wave 编号（有依赖关系）→ 串行执行
- 示例：某个 plan 文件依赖 `02-1` 完成后才能执行

---

#### 第四步：执行（execute）

**命令格式：** `execute <阶段编号>`（如 `execute 4`）

**执行逻辑：**
```
分析所有 plan 文件
    ↓
相同 wave 编号 → 并行：开启多个子Agent同时执行
不同 wave 编号 → 串行：按依赖顺序依次执行
```

这是最终的编码阶段，GSD 根据计划文件中的 wave 标记自动决定哪些任务并行、哪些串行，既提升了编码效率，又减少了主Agent上下文占用。

---

#### 第五步：验收（validate）

**命令格式：** `gsd validate <阶段编号>`（如 `gsd validate 2`）

**注意：** 这里的验收与传统"写测试用例"不同，是一种**人为参与的交互式验收**。

**两种验收方式：**

**方式一：人为参与验收（UIT - User Interaction Testing）**
- GSD 把所有需要测试的点列出来（如"点某个按钮是否出现某效果"）
- 用户自己去浏览器/App中测试
- 通过：输入 `pass`
- 不通过：把错误描述发给 AI，AI 修复后继续

**方式二：自动化测试（gsd test）**
- 命令：`gsd test`
- GSD 根据计划文件中的验证说明自动生成测试用例
- 也可以用 `gsd add test` 描述要测试的方法，AI 生成对应测试用例
- 生成后 AI 自动执行测试

> 优先推荐方式一（人为参与），方式二适合需要测试特定方法或流程时使用。

---

#### 其他重要命令

**添加阶段：** `gsd add`
- 为现有需求添加新阶段
- 添加后重新走讨论→计划→执行→验收的完整流程

**里程碑（Milestone）：**
- 适用于较大迭代，相当于一个标记点
- 内部执行过程与 `new project` 类似

**快速命令：** `quick`
- 适合小功能快速修改
- 不走完整的 discuss→plan→execute→validate 流程
- 直接修改代码，但会生成一个 quick 文件方便后续追溯
- 使用场景：简单的 UI 调整、小 bug 修复等

### 五、GSD 技术实现原理（源码学习价值）

GSD 的源码设计有很强的学习价值：

- 所有脚本/命令文件用**类 XML 结构**编写
- 命令本身只是**触发器**，真正的执行逻辑在根目录的 `workflow` 文件中
- 示例：执行 `new project` 命令 → 触发 → 根目录 workflow 中对应的工作流逻辑

> 强烈建议下载源码，用 AI 工具与其对话，深入理解设计原理。

### 六、GSD 的优缺点与适用场景

#### 缺点

| 缺点 | 说明 |
|------|------|
| **Token 消耗大** | 生成大量文件、开启多个子代理，均需消耗大量 Token |
| **用户参与空间少** | 完全通过对话询问方式构建文档，难以临时插入干预 |
| **需要精准区分使用场景** | 用错场景整个流程会非常漫长 |
| **学习成本高** | 30+ 个命令，需要深入学习才能熟练使用 |
| **严重依赖模型速度** | 模型生成文档和代码慢时，整体开发周期极长 |
| **"垃圾进垃圾出"风险** | GSD 完全遵循用户描述执行，用户需求不清晰则输出也不清晰 |

#### 适用场景

**适合使用 GSD 的情况：**
- 中大型项目
- 需要持续迭代的功能
- 团队本身有良好的开发流程和规范
- 对需求有清晰、完整的描述能力

**不适合使用 GSD 的情况：**
- 简单的小功能修改（用 `quick` 命令或直接对话即可）
- 需求尚不清晰、处于探索阶段
- 对模型速度要求较高、时间紧张

### 七、最常用的5个核心命令

在 GSD 的 30+ 命令中，日常使用只需掌握这5个：

| 命令 | 作用 |
|------|------|
| `new project` | 初始化项目，生成全局文件 |
| `discuss <n>` | 讨论第 n 阶段需求，生成 context.md |
| `plan <n>` | 为第 n 阶段生成具体执行计划 |
| `execute <n>` | 执行第 n 阶段（自动并行/串行） |
| `validate <n>` | 验收第 n 阶段功能 |

## 关键术语表

| 术语 | 解释 |
|------|------|
| **GSD** | Get Stuff Done，一个开源的规范驱动AI编程工作流，GitHub约3万stars |
| **规范驱动工作流** | 通过预定义的命令和文件规范来约束AI的行为和输出，而非依赖临时对话 |
| **子代理（Sub-Agent）** | 独立执行单个计划任务的AI实例，由主Agent创建和管理，上下文相对隔离 |
| **Wave 编号** | plan 文件中用于决定并行/串行关系的标识，相同编号的计划可并行执行 |
| **project.md** | GSD 全局文件之一，记录项目描述、主要功能和 MVP 目标 |
| **roadmap.md** | GSD 全局文件之一，将需求划分为多个有序阶段的路线图 |
| **state.md** | GSD 全局文件之一，追踪每个阶段的完成百分比，实现跨会话状态恢复 |
| **context.md** | 讨论阶段的产物，记录与用户确认后的需求细节 |
| **plan 文件（.plan）** | 计划阶段的产物，包含具体执行步骤、wave 编号和依赖关系 |
| **UIT（User Interaction Testing）** | 用户参与测试，由 AI 列出测试点，用户手动验证并反馈结果 |
| **map code base** | GSD 命令，用于扫描已有代码库，生成技术栈和架构概况文件 |
| **quick 命令** | GSD 提供的轻量级修改模式，不走完整流程，适合小功能修改 |
| **垃圾进垃圾出（GIGO）** | 若用户需求描述不清晰，GSD 会严格按不清晰的描述执行，输出同样不可靠 |

## 总结

GSD 的核心价值在于**把软件开发的工程规范固化进工作流命令**，通过强制性的文档体系和子代理机制，让 AI 编程从"随机漫步"变成"有轨道运行"。它的适用人群是那些有明确需求、追求可控性和可追溯性的开发者。

**使用 GSD 的关键收获：**
- 理解了"上下文臃肿"是 AI 编程质量下降的根本原因，子代理是解决方案
- 持久化文档体系（project.md + roadmap.md + state.md）是跨会话状态管理的标准答案
- Wave 编号机制是一种简洁优雅的并行/串行决策方案
- 人为参与的交互式验收（UIT）比纯自动化测试更适合前期功能验收

**注意点：**
- GSD 对需求描述的质量要求极高，务必在 `new project` 阶段提供详尽的 PRD
- 小功能修改优先用 `quick` 命令，不要用完整流程，否则时间和 Token 消耗极高
- 学习 GSD 最好的方式是下载源码，用 AI 工具与其对话，理解每个命令背后的工作流设计
