## 1. MMO 基本概念

### 1.1 什么是 MMO？

**MMO（Massive Multiplayer Online）**是指大量玩家同时在一个在线世界中连接、互动的游戏形式。

常见分类：
- **MMORPG**：大型多人在线角色扮演游戏（如 Final Fantasy XIV、魔兽世界）
- **MMOFPS**：大型多人在线第一人称射击游戏

> MMO 的本质是构建一个**持续存在的虚拟世界（Persistent Virtual World）**，玩家可以在其中感知彼此、互动交易。

### 1.2 MMO vs 传统多人游戏（Session-based）

| 维度 | Session-based（如 MOBA 战局） | MMO 持久世界 |
|------|-------------------------------|--------------|
| 世界持续性 | 会话结束即消失 | **7×24 小时持续运行** |
| 玩家规模 | 通常 2~100 人 | 数万~数百万同时在线 |
| 系统复杂度 | 较低（单纯战斗逻辑） | 极高（包含社交、交易、物流等） |
| 数据量级 | 会话内临时数据 | 海量持久化数据（玩家档案、道具、日志等） |

### 1.3 MMO 的历史渊源

- 最早的多人在线游戏是**基于文字的 MUD（Multi-User Dungeon）**
- 玩家通过文字指令探索房间、与他人互动（"我在桌上捡起了一个碗"）
- 现代 MMO 已从文字演变为高度丰富的 3D 虚拟世界

## 2. MMO 整体架构三层模型

一个 MMO 游戏服务器端的核心架构可以分为三层：

```
玩家（Client）
    ↓ 网络连接
┌─────────────────────────────────────┐
│           连接层（Link Layer）         │
│  Login Server  │  Gateway Server     │
└─────────────────────────────────────┘
    ↓ 内部通信（RPC / Message Queue）
┌─────────────────────────────────────┐
│           服务层（Service Layer）      │
│  Character  │ Trading │ Social      │
│  Server     │ System  │ System      │
│  Matchmaking│ Game    │ Mail Server │
│  Server     │ Server  │ ...         │
└─────────────────────────────────────┘
    ↓ 数据读写
┌─────────────────────────────────────┐
│       数据存储层（Data Layer）          │
│  关系数据库  │  NoSQL DB  │ 内存 DB   │
│ (MySQL)    │(MongoDB)  │ (Redis)   │
└─────────────────────────────────────┘
```

## 3. 连接层：Login Server 与 Gateway

### 3.1 Login Server（登录服务器）

**作用**：玩家进入游戏的**第一道门**，负责身份验证。

流程：
1. 玩家客户端通过**HTTPS** 加密信道连接 Login Server
2. 提交账号密码，Login Server 验证合法性
3. 验证通过后，下发**Token（令牌）**
4. 玩家使用 Token 连接 Gateway Server

> Login Server 与 Gateway 分离设计的原因：身份验证逻辑与游戏服务逻辑**职责分离**，Login Server 压力小且安全性高。

### 3.2 Gateway Server（网关服务器）

**Gateway 是 MMO 最核心的基础设施之一。**

#### 核心定位：防火墙 + 统一入口

```
[玩家 Client]  ←→  [Gateway Server]  ←→  [后端所有 Server]
                    （唯一对外暴露）       （对外完全隐藏）
```

**玩家永远只与 Gateway 通信，后端服务完全不对外暴露。**

#### Gateway 的主要职责

| 职责 | 描述 |
|------|------|
| **消息验证** | 验证所有来自客户端的消息合法性、数据格式有效性 |
| **连接管理** | 管理海量玩家 TCP/UDP 长连接的维持与断线处理 |
| **协议解析** | 解压缩、解密客户端数据包（CPU 密集操作） |
| **DDoS 防护** | 拦截异常流量，防止分布式拒绝服务攻击 |
| **消息路由** | 将不同类型消息分发给对应的后端 Service |
| **负载均衡** | 多个 Gateway 实例分担连接压力 |

#### 为什么 Gateway 通常要部署多个实例？

- 协议解析（解压缩、加密/解密）是**CPU 密集型操作**
- 随着玩家数量增长，单个 Gateway 无法承载
- 多 Gateway 实例 + 前置负载均衡器（LB），按玩家数量弹性扩容

> 类比：Gateway 就像酒店大堂的接待团队。玩家全部经过前台（Gateway）验证入住，后台的各部门（服务）不直接与客人接触。

## 4. 服务层：核心子系统

### 4.1 游戏大厅（Lobby Server）

**大厅是一种特殊的游戏模式**：
- 玩家可以互相看见、聊天，但不能进行战斗
- 作为**缓冲池（Buffer Pool）**，管理等待匹配的玩家
- 有时大厅本身也是一个 3D 场景（如 MMORPG 主城），也有虚拟大厅（纯逻辑状态）

### 4.2 角色服务器（Character Server）

**角色服务器是最核心的服务之一**，负责存储和管理**玩家的全量数据**。

玩家角色数据涵盖：
- 基础属性（等级、经验、血量）
- 背包道具（可达数 MB 甚至数十 MB）
- 任务进度、技能列表
- 历史操作记录

**为什么需要专用 Character Server？**

若角色数据分散在各个业务 Server（战斗服、聊天服、交易服）中：
- 数据碎片化，难以保证**一致性**
- 查询复杂，无法快速获取玩家完整状态

```
[战斗 Server] → 查询玩家装备    ─┐
[交易 Server] → 查询玩家背包    ─┤ → [Character Server]（单一数据源）
[社交 Server] → 查询玩家好友    ─┘
```

### 4.3 交易系统（Trading System）

交易系统具有**强金融属性**，对数据安全要求极高。

核心要求：**原子性（Atomicity）与可回滚（Rollback）**

```
玩家A 卖出道具 + 玩家B 支付金币  →  原子事务（Transaction）
```

若在事务中途发生以下异常：
- 客户端断线
- 服务器崩溃
- 网络超时

**必须保证数据回滚**，不能出现"道具消失了但金币也没转移"的情况。

> 道具在网游中具有真实经济价值，交易数据异常可能产生**法律责任**。

### 4.4 社交系统（Social System）

包含功能：
- 即时聊天（频道聊天、私聊）
- 邮件系统（异步消息）
- 好友管理（添加/删除/拉黑）
- 组队/公会管理
- 频道分层（世界频道、区域频道、队伍频道）

**为什么社交系统需要拆分多个服务？**

若所有社交功能在单一服务器上运行：
- 大量玩家同时聊天时，单进程 CPU/IO 直接打满
- 邮件服务器通常需要独立部署（峰值压力极高，如新年活动全服发邮件）

### 4.5 匹配系统（Matchmaking System）

**目标**：为玩家匹配**技能水平相近、网络延迟低**的对手。

匹配维度（以全球匹配射击游戏为例）：

| 维度 | 说明 |
|------|------|
| **天梯分（MMR / ELO）** | 技能水平接近，游戏公平性高 |
| **网络延迟（Ping）** | 地理位置接近，游戏体验好 |
| **等待时间** | 避免高分段玩家等待过久（适当放宽匹配范围） |
| **设备/平台** | 跨平台游戏考虑输入设备公平性 |

匹配策略通常是多维度加权优化问题，实际实现中会随等待时间动态**放宽匹配条件**。

## 5. 数据存储层

### 5.1 数据存储的三大类型

MMO 的数据存储通常同时采用三种类型的数据库：

#### 类型一：关系数据库（Relational Database）

代表：MySQL、PostgreSQL

- 用于存储**核心结构化数据**：玩家档案、道具信息、任务状态
- 支持**ACID 事务**，保证交易数据安全
- 挑战：海量并发写入时性能瓶颈；需要分库分表（Sharding）

**在 MMO 中的使用模式**：
- 分布式数据库架构：同时写入多个并行表，提高写入效率
- 冷热数据分离：活跃玩家数据（热表）与历史归档数据（冷表）分开
- 灾难备份（Disaster Recovery）

#### 类型二：非关系数据库（NoSQL）

代表：MongoDB、Cassandra、Redis（也可作为 NoSQL）

- 用于**临时 Game State、日志（Log）数据**
- 一个大型 MMO **每天可产生 TB 级日志数据**
- NoSQL 的优势：
  - 写入速度极快（无严格 Schema 约束）
  - 读写更轻量，适合流水式写入

**典型使用场景**：
- 玩家行为日志（用于客服找回道具、行为分析）
- 临时 Game State 缓存
- 事件流水记录

> 案例：玩家的道具丢失时，客服能找回来，是因为所有操作都记录在非关系数据库的日志里，可以回溯查询。

#### 类型三：内存数据库（In-Memory Database）

代表：Redis、Memcached

- 用于**服务间共享的中间数据、高频读写数据**
- 不频繁读写磁盘，速度比磁盘型数据库快 100~1000 倍
- 典型用途：
  - 在线玩家状态（Session 信息）
  - 排行榜数据
  - 多个服务共享的临时状态（如副本状态）

```
多个 Game Server 进程
    ↓           ↓           ↓
    └───────────┴───────────┘
              ↓
    [Redis 内存数据库]（共享中间数据）
```

### 5.2 数据架构设计原则

> 王希的建议：**设计 MMO 架构时，从数据开始思考。**

把数据模型想清楚，上层各个服务的结构也就自然清晰了。

| 数据类型 | 存储方案 | 理由 |
|----------|----------|------|
| 玩家核心档案（装备、等级） | 关系数据库 | ACID 事务保证 |
| 游戏日志、行为流水 | NoSQL | 大量写入，弱查询需求 |
| 服务间共享临时数据 | 内存数据库 | 高频读写，低延迟 |
| 实时 Game State | 内存数据库 / 进程内存 | 读写性能优先 |

## 6. 分布式系统架构

### 6.1 为什么需要分布式？

当玩家规模超过单服务器承载极限时：

$$\text{单服务器能力} < \text{玩家数量} \times \text{每玩家计算量}$$

以 Character Server 为例：
- 单进程管理 1 万玩家 → 可接受
- 10 万玩家在线时 → CPU 打满，必须**拆成多个进程（分布式）**

### 6.2 分布式系统的核心挑战

#### 挑战一：负载均衡（Load Balancing）

将玩家请求均匀分配到多个 Server 实例。

**一致性哈希（Consistent Hashing）** 是解决动态扩缩容下负载均衡的关键算法：

原理：
1. 将所有服务器映射到一个**环形哈希空间**（0 ~ $2^{32}-1$）
2. 将玩家 ID 也映射到相同的哈希空间
3. 每个玩家由**顺时针方向最近的服务器**负责

$$\text{server}(player) = \text{哈希环上 hash(playerID) 顺时针第一个服务节点}$$

优势：当增减服务器时，**只有少量玩家需要迁移**，不影响大多数玩家。

**案例**：玩家 101 获得道具，需要写入 Character Server。通过一致性哈希，快速确定该玩家归属哪个 Character Server 实例。

#### 挑战二：幂等性（Idempotency）

分布式网络中消息可能**重复发送**（网络重试、超时重发），需要保证：

> 同一消息处理多次的结果，与处理一次完全相同。

实现方式：
- 每个请求携带唯一的**请求 ID（Request UUID）**
- 服务端记录已处理的请求 ID，重复请求直接返回之前的结果

#### 挑战三：数据一致性（Data Consistency）

多个分布式服务访问同一数据时，需要防止：
- **竞争条件（Race Condition）**
- **死锁（Deadlock）**
- **脏读（Dirty Read）**

常用方案：分布式锁（如基于 Redis 的 Redlock）、版本号乐观锁（Optimistic Lock）。

#### 挑战四：Bug 级联放大

分布式系统中，一个低频 Bug 可能引发**全系统震荡**：

```
Service A 遇到 Bug → 产生异常数据
    → 传递给 Service B → B 将错误放大 → 传递给 Service C
    → C 通过 RPC 又回调 A → 无限震荡循环
    → 服务器性能急剧下降
```

**防范措施**：
- 每个服务的边界做**数据校验（Validation）**，拒绝异常数据
- 熔断器（Circuit Breaker）：检测到异常率超过阈值时自动中断调用链
- 监控 RPC 调用频次，发现异常高频调用立即告警

### 6.3 分布式系统的优势

- **弹性扩容（Elastic Scaling）**：根据在线玩家数量动态增减服务实例
- **高可用（High Availability）**：单个服务挂掉不影响整体
- **独立部署**：各服务可以独立更新、独立扩容

## 7. 大规模 MMO 扩展性设计

### 7.1 分服（Server Partition）与分线（Channel）

#### 分服（Realm / Server）

最传统的 MMO 扩展方式：将玩家分配到不同的**独立服务器（大区/服）**。

- 同一服务器内的玩家可以互相看见、互动
- 不同服务器的玩家完全隔离（不同世界）
- 优点：实现简单，单服规模可控
- 缺点：玩家社区被切割，无法全球一服

#### 分线（Channel / Instance）

同一大区下的**虚拟副本世界**：
- 同一地图同时有多个"线"（Channel）在运行
- 每条线有独立的玩家和实体
- 玩家可以主动切换到不同的线
- 通常用于**缓解热点区域拥挤**（如新手村、主城）

### 7.2 无缝大世界（Seamless Open World）

现代 MMO 追求**无加载界面的无缝世界**（如 EVE Online、星战前夜）。

**核心实现**：基于地理区域的分布式服务器（见第十九节上 Zone 分区设计）

- 地图划分为若干 **Zone（区域）**，每个 Zone 由独立进程管理
- 玩家跨越 Zone 边界时，**透明地从一个进程迁移到另一个进程**
- 边界区域的实体被**两侧 Zone 服务器同时跟踪**（Ghost/Shadow Entity）

### 7.3 游戏服务器（Game Server / Battle Server）

游戏服务器是负责**实际战斗/游戏逻辑**的计算节点。

主要职责：
- 运行物理模拟、AI 逻辑、战斗运算
- 管理场景内所有实体的状态
- 将状态变化广播给场景内的玩家客户端

与 Character Server 的分工：
- **Game Server**：负责实时运算和状态广播（生命周期短，以战斗/场景为单位）
- **Character Server**：负责持久化角色数据（长期存在）

## 8. 兴趣管理（AOI）

### 8.1 为什么需要 AOI？

在 MMO 大地图中，若**广播所有玩家的状态给所有人**：

$$\text{每秒传输量} = N \times (N-1) \times \text{每帧状态大小} \times \text{帧率}$$

当 $N = 10000$，每帧状态 100 bytes，20Hz 更新时：

$$10000 \times 9999 \times 100 \times 20 = \sim 200 \text{ GB/s}$$

这显然不可能。

**核心原则：玩家只需要关心自己能感知到的范围内的信息（Area of Interest）。**

### 8.2 AOI 的定义

**AOI（Area of Interest，感知区域）**：定义了每个玩家/实体应该收到哪些其他实体的状态更新。

只有当实体 B 进入实体 A 的 AOI 范围时，才开始向 A 同步 B 的状态。

### 8.3 AOI 算法：九宫格（Grid-based / Cross AOI）

**最简单且常用的 AOI 实现：将地图划分为等大小的格子。**

```
┌───┬───┬───┬───┬───┐
│   │   │   │   │   │
├───┼───┼───┼───┼───┤
│   │ X │ O │   │   │   X = 玩家, O = 感知范围
├───┼───┼───┼───┼───┤       九宫格 = 3×3 格子范围
│   │ O │ X │ O │   │
├───┼───┼───┼───┼───┤
│   │ O │ O │ O │   │
├───┼───┼───┼───┼───┤
│   │   │   │   │   │
└───┴───┴───┴───┴───┘
```

实现逻辑：
- 每个玩家维护一个**格子坐标**
- 玩家的 AOI 范围 = 以自身格子为中心的 $3 \times 3$（或更大的 $n \times n$）邻域
- 当玩家移动到新格子时，计算**进入/离开 AOI 的实体集合**变化

```
玩家移动 → 计算新旧 AOI 格子集合差集
  进入新格子集合的实体 → 开始订阅同步
  离开旧格子集合的实体 → 取消订阅同步
```

优点：实现简单，查询效率 $O(1)$（格子查找）
缺点：精度较粗（格子越大，AOI 边界越不准确）

### 8.4 AOI 算法：灯塔法（Beacon / Publish-Subscribe）

**灯塔法**将地图分为若干固定的**灯塔格子（Beacon Cell）**，采用发布-订阅模式：

```
每个灯塔格子有：
    Watcher 列表（订阅者）= 在其 AOI 内包含该格子的玩家
    Marker 列表（发布者）= 当前在该格子内的实体
```

当实体 B 移动进入格子 G 时：
- 向 G 的所有 Watcher 广播"实体 B 进入"事件

当实体 A 移动时：
- A 从旧格子的 Watcher 列表中**退订**
- A 加入新格子的 Watcher 列表中**订阅**

$$\text{订阅操作开销} = O(\text{灯塔格子数量})$$

与九宫格的对比：
| | 九宫格 | 灯塔法 |
|--|------|-------|
| 查询方式 | 基于坐标直接查格子邻域 | 发布-订阅事件驱动 |
| 更新效率 | 移动时需重新计算 AOI 格子列表 | 事件驱动，按需推送 |
| 实现复杂度 | 低 | 中 |
| 适用场景 | 均匀分布、低密度 | 高密度、不均匀分布 |

### 8.5 AOI 的参数设计

- **AOI 半径**：过小 → 玩家看到其他角色"凭空出现"；过大 → 同步数据量大
- **动态 AOI**：根据玩家速度、游戏类型动态调整感知范围大小
- **优先级分级**：AOI 内的实体按距离分优先级，近处实体更新频率高，远处更新频率低

## 总结

本节课（下半部分）系统讲解了**MMO 大规模在线游戏的服务器架构设计**：

### 架构分层

MMO 服务器分为**三层**：连接层（Login + Gateway）、服务层（Character/Trading/Social/Matchmaking）、数据存储层（关系DB + NoSQL + 内存DB）。

### Gateway 的核心地位

Gateway 是 MMO 安全和稳定的**第一道防线**，承担消息验证、连接管理、DDoS 防护等职责。玩家永远只与 Gateway 通话，后端服务完全隐藏。

### 数据从设计开始

> "设计 MMO 架构时，从数据开始思考。数据模型想清楚了，系统结构自然清晰。"

三类数据库各司其职：关系DB保证事务安全、NoSQL处理海量日志、内存DB支撑高频中间数据。

### 分布式是规模化的必由之路

当玩家规模超过单机承载时，必须引入分布式架构。核心要解决：**一致性哈希负载均衡**、**幂等性**、**数据一致性**、**Bug 级联防护**。

### AOI 是带宽优化的核心

通过 AOI 将广播复杂度从 $O(N^2)$ 降低到 $O(N \times K)$（$K$ 为 AOI 内平均实体数），使大规模 MMO 在带宽约束下成为可能。

### 关键设计权衡

| 权衡点 | 选项 A | 选项 B |
|--------|--------|--------|
| 架构简单性 vs 扩展性 | 单体服务器（简单）| 分布式（可扩展）|
| 一致性 vs 可用性 | 强一致（慢）| 最终一致（快）|
| AOI 精度 vs 开销 | 小 AOI（精确但可能信息缺失）| 大 AOI（覆盖全但带宽大）|
| 实时性 vs 平滑性 | 外插值（实时但跳变）| 内插值（平滑但有延迟）|
