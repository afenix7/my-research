# 第三节：如何构建游戏世界

## 概述

本节课是 GAMES104《现代游戏引擎：理论与实践》第三讲，核心问题是：游戏世界由什么构成？如何描述这些东西？它们之间如何组织和调动？这是游戏引擎最核心的设计问题。

---

## 1. 游戏世界的组成元素

游戏世界可以被拆解为以下几类对象：

### 1.1 动态物（Dynamic Game Object）
- 可以移动、可被操控的交互物体
- 例如：坦克、无人机、小兵、导弹
- 这是玩家最容易注意到的一类

### 1.2 静态物（Static Game Object）
- 放置在场景中、相对固定的装饰或功能物体
- 例如：瞭望塔、机棚、建筑、掩体
- 构成游戏场景的骨架

### 1.3 环境系统
- **地形（Terrain）**：无限延伸的大地，是所有物体的支撑托盘，通常作为独立系统实现
- **天空（Sky / TOD）**：Time of Day 系统，表现日夜变化、阴天晴天、云层动效
- **植被（Foliage）**：数量巨大，兼具静态固定和动态响应（风吹、爆炸交互）的特点

### 1.4 不可见的逻辑物体
- **Trigger Box（触发区域）**：走入区域时触发事件，例如积分区域、BOSS 触发点
- **空气墙（Blocker）**：阻止玩家越界，例如战地的 War Zone 边界
- **战区规则对象**：游戏规则本身也可以被抽象成一个物体

---

## 2. GameObject（GO）——万物皆对象

### 2.1 核心抽象

现代游戏引擎将上述所有类型（动态物、静态物、地形、触发体、规则对象）统一抽象为 **GameObject（GO）**。

> 游戏世界 = 管理好一系列 GameObject

**GO** 是行业缩写，与 Go 语言无关。

### 2.2 物体的两类描述维度

描述任何一个 GameObject，可以从两个维度入手：

| 维度 | 含义 | 举例 |
|------|------|------|
| **Property（属性）** | 静态数据，描述物体是什么 | 位置 `Vector3`、血量 `float [0,100]`、油量 `float [0,1]` |
| **Behavior（行为）** | 动态逻辑，描述物体能做什么 | `move()`、`patrol()`、`attack()` |

### 2.3 面向对象设计的局限性

最早期的引擎直接用面向对象（OOP）的继承体系描述世界：

```cpp
class Drone {
    Vector3 position;
    float   health;
    float   fuel;
    void    move();
    void    patrol();
};

class CombatDrone : public Drone {
    int  ammo;
    void attack();
};
```

**问题：** 随着游戏复杂度提升，继承体系出现多重继承困境。例如"水陆两栖坦克"：
- 坦克派生自"车辆"
- 船派生自"船只"
- 水陆两栖坦克的祖先到底是谁？

这是经典的 **菱形继承问题（Diamond Inheritance Problem）**。

## 3. 组件化（Component-Based Architecture）

### 3.1 核心思想

将 GameObject 的行为拆分成一个个独立的 **Component（组件）**，像积木一样自由组合，而不依赖继承体系。

类比：射击游戏中的枪械自定义——加三倍镜变狙击枪，加消音器变微冲，本质是组件的组合。

### 3.2 典型组件结构

一个巡逻无人机 GO 的组件构成：

```
GameObject: 无人机
├── TransformComponent    -- 空间位置、旋转、缩放
├── ModelComponent        -- 几何外形（三角网格）
├── MotorComponent        -- 运动属性（最大速度、加速度、惯性）
├── AnimationComponent    -- 动画状态机
├── PhysicsComponent      -- 物理碰撞体
├── AIComponent           -- 巡逻/追踪 AI 行为
└── HealthComponent       -- 血量管理
```

将其升级为察打一体无人机，只需替换 AI 模块、增加 CombatComponent：

```
GameObject: 察打一体无人机
├── TransformComponent
├── ModelComponent
├── MotorComponent
├── AnimationComponent
├── PhysicsComponent
├── AttackAIComponent     -- 替换普通 AI，增加索敌/瞄准
├── CombatComponent       -- 武器系统
└── HealthComponent
```

### 3.3 ComponentBase 基类设计

所有组件派生自一个公共基类：

```cpp
class ComponentBase {
public:
    virtual void tick(float deltaTime) = 0;
    // ... 其他生命周期接口
};

class TransformComponent : public ComponentBase { ... };
class AnimationComponent : public ComponentBase { ... };
```

关键点：每个 Component 都必须实现 `tick()` 函数。

### 3.4 Unity / Unreal 中的体现

- **Unity**：对象（GameObject）下挂载各种 Component，可通过编辑器增删，可编写脚本
- **Unreal**：`UObject` 是更底层的内存管理基类（类似 C# 的 `object`），`Actor + Component` 才对应我们说的 GO 概念；Unreal 使用 C++ 反射机制管理 Component 的消息注册

## 4. Tick 机制——让世界动起来

### 4.1 Tick 的基本概念

游戏引擎每隔固定时间（如 $\frac{1}{30}$ 秒 = 33ms）让世界往前走一步，这个步骤称为 **Tick**。

> 类比：物理学中的普朗克时间（Planck Time）是宇宙的最小时间单元，Tick 是游戏世界的最小时间单元。

每个 Tick 中，对所有 GO 的所有 Component 依次调用 `tick(deltaTime)`：

$$\text{WorldTick}(t) \rightarrow \forall \text{GO}_i,\ \forall \text{Comp}_j \in \text{GO}_i : \text{Comp}_j.\text{tick}(\Delta t)$$

### 4.2 基于对象的 Tick vs. 基于系统的 Tick

| 方式 | 逻辑 | 优点 | 缺点 |
|------|------|------|------|
| **对象 Tick（直觉）** | 对每个 GO，依次 tick 其所有组件 | 直觉，符合人类思维 | 数据分散，缓存不友好 |
| **系统 Tick（反直觉）** | 对每种 Component 类型，批量 tick 所有实例 | 数据局部性好，可批处理，高效 | 设计复杂 |

**系统 Tick 的类比**：汉堡流水线——专人烤面包，专人烤牛肉，专人洗菜，最终统一合成，效率远高于每人独立完成全套流程。

现代游戏引擎为追求效率，逐渐转向**按 System / 按 Component 类型进行批量 Tick**（这也是 ECS 架构的核心思想）。

### 4.3 Tick 的变体函数

真实引擎中 Tick 并非只有一个函数，常见变体：
- `PreTick` / `Tick` / `PostTick`：用于处理 Component 之间的时序依赖

## 5. Component 通信——事件机制

### 5.1 问题：GO 之间如何交互？

当炮弹命中敌人时，炮弹 GO 需要通知敌人 GO 扣血。最原始的方案是**直接调用**（Hot Code）：

```cpp
// 炮弹爆炸时，硬编码判断周边所有物体类型
if (target is Human)   target.health -= damage;
if (target is Tank)    target.health -= damage;
if (target is Stone)   /* 无影响 */
```

**问题**：随着游戏复杂度增加，这种硬编码难以维护，且产生严重的 $O(N^2)$ 性能问题（N 个物体相互检测）。

### 5.2 事件机制（Event System）

采用**邮局模型**解耦 GO 之间的通信：

1. GO 不直接调用对方，而是向"邮局"（事件系统）发送 Event
2. 事件系统在合适时机将消息派发到订阅了该事件的 GO/Component
3. Component 通过回调函数处理收到的 Event

```
炮弹 GO                 事件系统                  小兵 GO
    |                       |                          |
    |-- DamageEvent(100) -->|                          |
    |                       |-- dispatch DamageEvent ->|
    |                       |                          |-- HealthComp.OnDamage()
```

**Unity 中的实现**：通过字符串匹配 `SendMessage("OnDamage", data)` 触发回调

**Unreal 中的实现**：使用 C++ 反射机制 + Blueprint 中的 Delegate/Event，需要显式声明和绑定

### 5.3 消息时序与确定性（Determinism）

**核心问题**：在多线程并行 Tick 下，如果 GO 之间可以在同一帧直接互发消息，会产生逻辑歧义（Ambiguity）。

**问题举例**：A 在当前帧发出 Event 给 B，B 在同帧收到并回应 —— 但如果 B 也在同帧向 A 发出 Event，谁先谁后无法确定，结果不确定（Non-deterministic）。

**为什么需要确定性**：游戏回放（Replay）功能只记录输入，回放时重新执行游戏逻辑——如果逻辑不确定，相同输入无法复现相同结果。

**解决方案**：引入"邮局"（Event Queue），当帧内产生的所有 Event 先入队，统一在下一帧开始时派发，保证时序严格一致：

$$\text{Frame}_n \text{ 产生的 Events} \rightarrow \text{Frame}_{n+1} \text{ 开始时统一派发}$$

### 5.4 Component 依赖顺序

Component 之间存在依赖关系，Tick 顺序至关重要：

```
Motor.tick()        -- 计算新速度/位移
  ↓
Animation.tick()    -- 根据速度切换动画状态（走→跑）
  ↓
Physics.tick()      -- 根据动画骨骼更新碰撞体
```

如果顺序错误，会产生 1~2 帧的逻辑延迟（Lag），这是游戏中常见的"手感不好"问题根源。

## 6. 场景管理（Scene Management）

### 6.1 N² 问题

若场景中有 $N$ 个 GO，每次碰撞/事件检测都需要与其他所有物体比较：

$$\text{计算量} \approx N \times (N-1) \approx N^2$$

当 $N = 10000$ 时，计算量约为 **1 亿次**，每帧 30ms 内完全不可接受。

### 6.2 均匀网格划分（Grid）

最简单的方案：将世界划分为均匀格子，只检测邻近格子内的物体。

**优点**：实现简单，适合物体分布较均匀的小型场景
**缺点**：物体分布不均时浪费严重（大部分格子为空）

### 6.3 层次结构（Hierarchy）空间划分

类比地图分层管理：世界→国家→行省→城市→街区→街道

当事件发生时，只需在对应的小区域内搜索，大幅减少比较次数。

常见的层次空间划分结构：

| 数据结构 | 说明 | 适用场景 |
|----------|------|----------|
| **二叉空间划分树（BSP Tree）** | 用超平面递归切割空间，可沿墙面切割 | 室内场景（如 Quake） |
| **四叉树（Quadtree）** | 2D 空间递归四等分 | 2D 游戏，俯视角策略游戏 |
| **八叉树（Octree）** | 3D 空间递归八等分 | 三维场景 |
| **BVH（Bounding Volume Hierarchy）** | 从物体出发，用包围盒（AABB）自底向上合并 | **现代引擎首选**，适合动态场景 |

### 6.4 BVH 的工作原理

每个物体有一个 AABB（Axis-Aligned Bounding Box）包围盒，从叶节点向上逐层合并：

```
         [根节点 AABB]
        /             \
  [子树 AABB 1]   [子树 AABB 2]
   /     \           /     \
[A]     [B]        [C]     [D]
```

当进行碰撞检测或视锥裁剪时，从根节点向下递归：若当前节点的 AABB 不相交，则整个子树可以被跳过。

**时间复杂度**：从 $O(N^2)$ 降低至约 $O(N \log N)$

## 7. 高级话题：绑定与时序

### 7.1 物体绑定（Object Binding / Attachment）

当玩家坐上车辆时，玩家 GO 被"绑定"到车辆 GO 上：
- 车辆 GO 是父节点（Parent），先 Tick
- 玩家 GO 是子节点（Child），后 Tick，从父节点获取最新位置

### 7.2 组件循环依赖

真实引擎中 Component 之间存在复杂的依赖关系，可能形成循环：

```
Motor → Animation → Physics → (反过来影响 Motor？)
```

这种循环依赖是游戏引擎设计的核心难点，PreTick/PostTick 机制是处理这类问题的常见手段。

## 8. 工具层：世界编辑器

游戏引擎最终需要给艺术家和设计师使用，因此引擎的组织方式必须**直觉、易理解、可扩展**。

- **Unity 编辑器**：选中任意物体可看到其 Component 列表，可增删、可编写脚本
- **Unreal 编辑器（蓝图）**：通过 Blueprint 可视化地注册和响应 Event

组件化架构不仅是性能上的选择，也是**生产力工具设计**上的选择。

## 关键术语表

| 术语 | 全称 / 含义 |
|------|-------------|
| **GO** | Game Object，游戏对象，游戏世界的基本单元 |
| **Component** | 组件，描述 GO 某一方面能力的最小单元 |
| **Tick** | 游戏世界推进一步的函数，通常每帧调用一次 |
| **deltaTime** | 每帧耗时，用于保证帧率无关的物理计算 |
| **Event / Message** | 事件/消息，GO 之间解耦通信的机制 |
| **Determinism** | 确定性，相同输入必然产生相同输出 |
| **BVH** | Bounding Volume Hierarchy，包围盒层次结构 |
| **Quadtree** | 四叉树，2D 空间层次划分 |
| **Octree** | 八叉树，3D 空间层次划分 |
| **BSP** | Binary Space Partitioning，二叉空间划分 |
| **TOD** | Time of Day，日夜变化系统 |
| **Trigger Box** | 触发区域，不可见但可检测玩家进入的逻辑体 |
| **ECS** | Entity-Component-System，进阶架构（本课程后续讲解） |
| **GUID / UID** | 全局唯一标识符，用于唯一标记每个 GO |

## 总结

本节课建立了现代游戏引擎中"世界构建"的核心框架：

1. **万物皆 GameObject**：所有场景元素（动态物、静态物、不可见逻辑体）统一抽象为 GO
2. **组件化描述**：GO 由多个 Component 组合而成，每个 Component 负责一个功能维度，取代继承体系
3. **Tick 驱动**：通过定期 Tick 让世界动起来；现代引擎倾向于按 System 批量 Tick 以提升效率
4. **事件机制解耦**：GO 之间通过 Event Queue 通信，保证逻辑确定性，支持游戏回放
5. **层次空间管理**：用 BVH、四叉树等结构将场景查询从 $O(N^2)$ 降低至 $O(N \log N)$

这五个设计决策共同构成了现代商业引擎（Unity、Unreal 等）的基础骨架。
