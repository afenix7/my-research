# 第四节：游戏引擎中的渲染实践

## 概述

本节是 GAMES104 渲染系统四节课的第一讲，由王希主讲。课程定位明确：**游戏渲染是工程实践科学，而非理论图形学**。本节重点覆盖 GPU 硬件架构基础、渲染数据组织方式、可见性裁剪机制以及 Render Pass 的基本结构。

---

## 1. 游戏渲染 vs 理论图形学

### 1.1 四大核心挑战

| 挑战 | 描述 |
|------|------|
| **极度复杂性** | 场景中有数千对象（水体、植被、角色、云、地面），每种对象需要不同算法 |
| **硬件深度适配** | 必须在真实硬件（PC、主机、Switch）上高效运行，不能只追求算法正确性 |
| **帧率稳定性** | 无论场景简单复杂，必须保证稳定 30fps / 60fps / 120fps |
| **CPU 预算有限** | 渲染只能占用约 10%~20% 的 CPU 算力，其余留给游戏逻辑、网络等系统 |

### 1.2 关键区别

理论图形学（如辐射度算法）可以用几天时间渲染一帧，追求物理正确；游戏渲染必须在 **33ms（30fps）或 16ms（60fps）** 内完成全部渲染，这是一个极其严苛的实时预算约束。

> 游戏引擎的绘制系统是经过近 30 年工程迭代、被 heavily optimized 的软件系统，是实践科学。

## 2. GPU 硬件架构基础

### 2.1 SIMD 并行计算思想

GPU 与 CPU 的本质区别：

| 特性 | CPU | GPU |
|------|-----|-----|
| 核心数 | 少（4~16） | 极多（数千） |
| 擅长任务 | 复杂逻辑、分支 | 大量相同计算并行执行 |
| 设计目标 | 通用计算 | 数值密集型并行计算 |

GPU 的核心思想是 **SIMD（Single Instruction Multiple Data）**：用同样的代码，同时处理大量数据，每个计算单元访问自己的数据。这使得 GPU 的数学运算能力远超 CPU。

### 2.2 现代 GPU 架构（以 NVIDIA Fermi 架构为例）

```
显卡
└── GPC (Graphics Processing Cluster) × N    -- 图形处理集群
    └── SM (Streaming Multiprocessor) × M     -- 流式多处理器
        ├── CUDA Core × 32                    -- 执行加减乘除
        ├── SFU (Special Function Unit)       -- 执行 sin/cos 等特殊函数
        ├── Texture Unit                      -- 纹理采样硬件
        └── Shared Memory                     -- SM 内各核心共享缓存
```

**现代架构（Ampere / Hopper）新增：**
- **Tensor Core**：专用 AI 矩阵运算（支持 DLSS）
- **RT Core**：专用光线追踪加速

### 2.3 CPU-GPU 数据传输瓶颈

冯·诺依曼架构将计算与数据分离，由此产生严重的数据传输瓶颈：

```
CPU 主内存  <--- PCIe 总线（慢）--->  GPU 显存
```

**核心原则：数据单向传输（CPU → GPU），尽量不从 GPU 读回数据**

若需要 GPU 计算结果返回 CPU 再决策（Data Readback），会导致约半帧到一帧的 Latency，造成逻辑与画面不同步。

### 2.4 Cache（缓存）命中率

内存访问延迟对比：

| 存储层级 | 访问延迟 |
|----------|---------|
| 寄存器 | ~1 时钟周期 |
| L1 Cache | ~4 时钟周期 |
| 主内存 (RAM) | ~100~300 时钟周期 |

**Cache Hit（缓存命中）**：数据在缓存中，访问极快
**Cache Miss（缓存未命中）**：需要等待 100+ 时钟周期从内存读取，性能急剧下降

**结论**：渲染数据必须尽量排列在连续内存中，以提高 Cache 命中率。这也是 Mesh、Texture 等数据的内存布局设计原则。

## 3. 渲染数据组织

### 3.1 渲染对象的数据抽象

游戏场景中的每个可渲染 GameObject，需要向渲染系统提供以下数据：

```
Renderable Object
├── Mesh（网格）
│   ├── Vertex Buffer：顶点坐标 (x, y, z)
│   ├── Index Buffer：三角面索引
│   ├── Normal / UV / Tangent 等属性
│   └── Submesh 划分（不同材质区域）
├── Material（材质）
│   ├── Shader Program（着色器程序）
│   ├── 材质参数（Base Color、Roughness、Metallic 等）
│   └── Texture 引用列表
└── Transform（空间变换矩阵）
    ├── World Matrix（世界矩阵）
    └── 法线变换矩阵（逆转置）
```

### 3.2 Mesh 网格

**多边形网格（Polygon Mesh）** 是游戏中几何体的主要表示方式：

- 由三角形（Triangle）组成，因为三角形永远共面，适合 GPU 光栅化
- 顶点数据存储在 **Vertex Buffer Object（VBO）** 中
- 拓扑关系存储在 **Index Buffer Object（IBO）** 中

**Triangle Strip / Triangle Fan** 可减少重复顶点，但现代引擎多用 Indexed List。

### 3.3 Material 材质与 Texture 纹理

**Texture（纹理）** 是贴在 Mesh 表面的图像数据：

| 纹理类型 | 用途 |
|----------|------|
| Albedo Map（漫反射贴图）| 表面基础颜色 |
| Normal Map（法线贴图）| 模拟微观表面凹凸 |
| Roughness Map | 控制表面粗糙度 |
| Metallic Map | 控制金属度 |
| AO Map（环境光遮蔽）| 预计算环境光遮蔽 |
| Emissive Map | 自发光贴图 |

**Mipmap（多级纹理）**：为不同距离的物体预生成不同分辨率的纹理，降低远处纹理的采样频率，避免闪烁（Aliasing）：

$$\text{Level}_k = \frac{1}{2^k} \text{ 分辨率}$$

存储开销 $= 1 + \frac{1}{4} + \frac{1}{16} + \cdots \approx \frac{4}{3}$（额外 33% 存储）

**纹理压缩格式**：
- **DXT / BC（Block Compression）**：PC 平台标准，4:1 压缩比
- **ASTC**：移动端标准，可变压缩比

### 3.4 Shader 系统

Shader 是运行在 GPU 上的程序，描述如何计算顶点和像素的颜色：

**渲染管线中的主要 Shader 类型：**

```
CPU                          GPU Pipeline
提交 Draw Call  →  Vertex Shader → Rasterization → Pixel/Fragment Shader → Output
                   （顶点变换）     （光栅化）        （像素着色）
```

**Vertex Shader**：将模型空间顶点变换到裁剪空间

$$\mathbf{p}_{clip} = M_{proj} \times M_{view} \times M_{world} \times \mathbf{p}_{local}$$

**Pixel Shader（Fragment Shader）**：计算每个像素最终颜色，包括光照、纹理采样、材质计算等

**Shader 变体（Shader Permutation）** 问题：不同材质、不同渲染条件需要不同的 Shader 版本，数量可能爆炸性增长（成千上万个变体），是现代引擎管理的重大挑战。

## 4. 可见性裁剪（Visibility Determination）

在每帧渲染前，必须确定哪些物体是玩家摄像机能看见的，只提交这些物体给 GPU，减少无效计算。

### 4.1 视锥裁剪（Frustum Culling）

摄像机的可见空间是一个**视锥体（View Frustum）**：

```
        Near Plane
       /----------\
      /            \
     /              \
Camera              Far Plane
```

视锥体由 6 个平面定义：Near、Far、Left、Right、Top、Bottom。

**算法**：对每个物体的包围盒（AABB / Sphere），判断是否与视锥体相交：
- **完全在内**：可见，提交渲染
- **完全在外**：剔除
- **部分相交**：保留（保守策略）

结合 BVH（包围体层次结构），可从 $O(N)$ 降低为 $O(\log N)$ 的检测复杂度。

### 4.2 遮挡剔除（Occlusion Culling）

视锥裁剪只解决"在摄像机视野外"的情况，但视野内有大量物体被前景遮挡，仍然不需要渲染。

**常见方法：**

| 方法 | 原理 | 特点 |
|------|------|------|
| **软件 Occlusion Culling** | CPU 上用低精度深度图做遮挡测试 | 精确，CPU 开销 |
| **Hardware Occlusion Query** | GPU 渲染包围盒，查询是否通过深度测试 | 有 GPU Readback 延迟 |
| **Hi-Z Culling（Hierarchical Z）** | 利用 GPU 的 Hierarchical Depth Buffer | 高效，现代引擎主流 |
| **Portal System** | 手工标记遮挡关系，适合室内场景 | 精确但制作成本高 |

**Hi-Z（Hierarchical Z-Buffer）原理：**
构建多层级深度缓冲（类似 Mipmap），粗粒度快速排除被遮挡的大批物体。

### 4.3 背面剔除（Backface Culling）

对于封闭的 Mesh，朝向摄像机背面的三角面是不可见的，可直接丢弃，节省约 50% 的几何处理量。

通过三角形顶点绕序（Winding Order）判断正反面：
- **顺时针（CW）** 或 **逆时针（CCW）** 定义正面方向（引擎统一约定）

## 5. LOD（Level of Detail）机制

**核心思想**：远处的物体不需要高精度模型，使用更简单的版本以节省 GPU 算力。

### 5.1 静态 LOD（Discrete LOD）

同一个资产预制多个不同多边形数量的版本：

```
LOD 0（最近）：~10000 面    高精度
LOD 1         ：~2000 面
LOD 2         ：~500 面
LOD 3（最远）：~100 面     低精度
```

根据物体在屏幕上的**像素覆盖面积（Screen Size）**自动切换 LOD 级别。

**LOD 切换突变（Popping）问题**：通过 Alpha 抖动（Alpha Dithering）或距离过渡混合（Cross-fade）缓解视觉突变。

### 5.2 动态 LOD

- **Progressive Mesh（渐进网格）**：运行时动态简化网格
- **Nanite（Unreal Engine 5）**：基于微多边形（Micro Polygon）的虚拟化几何体，几乎无限细节

## 6. 渲染 Pass 组织

### 6.1 渲染管线（Render Pipeline）概览

现代游戏渲染并非一个单一的"全场景绘制"步骤，而是由多个 **Render Pass** 组成的管线：

```
Shadow Pass         -- 生成阴影贴图（从光源视角渲染深度）
    ↓
Depth Pre-Pass      -- 提前渲染场景深度（Early-Z）
    ↓
G-Buffer Pass       -- 延迟渲染：存储几何信息（Deferred Shading）
    ↓
Lighting Pass       -- 基于 G-Buffer 计算光照
    ↓
Transparent Pass    -- 正向渲染透明物体
    ↓
Post-Process Pass   -- 后处理（Bloom、DOF、TAA、Tone Mapping 等）
    ↓
UI Pass             -- 渲染 HUD / UI
```

### 6.2 前向渲染 vs 延迟渲染

| 模式 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **Forward Rendering（前向渲染）** | 每个物体直接计算所有光照 | 简单，支持透明、MSAA | 多光源开销 $O(N \times L)$ |
| **Deferred Rendering（延迟渲染）** | 先写 G-Buffer，再统一计算光照 | 多光源开销 $O(N + L)$，主流 3A 引擎选择 | 不支持透明、MSAA 困难 |
| **Forward+（Tiled Forward）** | 按屏幕 Tile 划分光源影响范围 | 兼顾两者优点，现代引擎趋势 | 实现复杂 |

### 6.3 Draw Call 优化

每次 CPU 调用 GPU 绘制一个物体称为一次 **Draw Call**。Draw Call 本身有固定开销（驱动层切换状态），因此减少 Draw Call 数量至关重要。

**常见优化方法：**

| 方法 | 原理 |
|------|------|
| **静态批处理（Static Batching）** | 将相同材质的静态物体合并为一个 Mesh |
| **动态批处理（Dynamic Batching）** | 运行时合并小的动态物体 |
| **GPU Instancing** | 同一个 Mesh 在不同位置绘制多次，只发一次 Draw Call |
| **Indirect Draw / MDI** | CPU 一次提交大量 Draw Call 参数，GPU 自行执行 |

## 关键术语表

| 术语 | 含义 |
|------|------|
| **Draw Call** | CPU 向 GPU 发出一次绘制命令 |
| **Vertex Buffer（VBO）** | 存储顶点数据的 GPU 缓冲区 |
| **Index Buffer（IBO）** | 存储三角面索引的 GPU 缓冲区 |
| **Shader** | 运行在 GPU 上的着色程序 |
| **Mipmap** | 预生成的多级分辨率纹理 |
| **Frustum Culling** | 视锥裁剪，剔除视野外物体 |
| **Occlusion Culling** | 遮挡剔除，剔除被前景遮挡的物体 |
| **Backface Culling** | 背面剔除，丢弃朝向相机背面的三角形 |
| **LOD** | Level of Detail，细节层次，远处用低精度模型 |
| **G-Buffer** | 延迟渲染中存储几何信息的缓冲区集合 |
| **SIMD** | Single Instruction Multiple Data，GPU 并行计算基础 |
| **SM（Streaming Multiprocessor）** | GPU 上的流式多处理器，包含多个 CUDA Core |
| **Cache Hit / Miss** | 缓存命中/未命中，影响计算效率 |
| **Render Pass** | 渲染管线中的一个独立阶段 |
| **PBR（Physically Based Rendering）** | 基于物理的渲染，下一讲重点内容 |

## 总结

本节建立了游戏渲染系统的工程认知框架：

1. **硬件约束是第一原则**：GPU 的 SIMD 并行架构、CPU-GPU 带宽限制、Cache 局部性要求，决定了渲染数据的组织方式
2. **数据管理是渲染核心**：Mesh、Material、Texture 的高效组织是渲染系统性能的基础，必须考虑内存布局对 Cache 的影响
3. **裁剪先于渲染**：Frustum Culling + Occlusion Culling + Backface Culling 三级裁剪，将提交到 GPU 的数据量降至最低
4. **LOD 是性能与质量的平衡**：根据屏幕占比自动切换细节层次，保证帧率预算
5. **Pass 化组织渲染流程**：将渲染拆分为 Shadow Pass、G-Buffer Pass、Lighting Pass、Post-Process Pass 等，现代引擎普遍采用延迟渲染或 Forward+ 架构

后续三节课将深入讲解：光照模型（PBR/IBL）、材质系统、阴影技术、全局光照、后处理，以及实时渲染的前沿技术。
