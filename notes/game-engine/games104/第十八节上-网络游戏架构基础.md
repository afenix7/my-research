## 网络游戏的核心挑战

网络游戏相对于单机游戏的本质差异：让任意玩家在任意时候能够联机共同体验。然而这带来了若干根本性挑战。

### 挑战一：世界一致性（Consistency）

> 游戏引擎的所有复杂度，最初只考虑一个人的情况。一旦加入多人，复杂度剧增。

当 100 个玩家联网进入虚拟世界时，每个人看到的世界状态都是不同的（由于网络延迟、丢包等）。如何在存在延迟与不可靠传输的前提下，让所有玩家的世界保持**事件一致性**，是网络游戏最核心的问题。

- 每个客户端都是一个独立的"宇宙副本"
- 需要协调机制确保各副本中发生的事件不冲突

### 挑战二：传输不可靠性（Unreliability）

网络传输本质上是不可靠的：

- **丢包（Packet Loss）**：数据包在传输途中丢失
- **延迟（Latency）**：数据到达存在时间差
- **拥塞（Congestion）**：网络通道过载
- **断线/重连（Disconnect/Reconnect）**：连接中断后需要状态恢复

与上帝视角不同——"上帝有无限带宽，且不丢包"——而我们的游戏网络环境极不稳定。

### 挑战三：安全与反作弊（Anti-Cheat）

> 网络游戏本质上是一个巨大的经济系统。

单机游戏中的内存修改（如锁定血量、金钱）无伤大雅；但在多人游戏中，任何一方作弊都会破坏游戏公平性。常见威胁：

- **内存篡改**：修改本地游戏状态
- **DDoS 攻击**：对服务器发起洪水攻击
- **数据包伪造**：发送非法数据包
- **个人信息泄露**：IP 地址、账号信息暴露

### 挑战四：多样性（Diversity）

现代在线游戏需要支持异构设备和平台：

- 客户端：iOS、Android、PC（Windows/Linux/macOS）
- 服务器：Linux-based 服务器集群
- 所有设备需运行同一套逻辑并能互联

此外，**热更新（Hot Update）**也是必须解决的问题：游戏上线后需在不停服的情况下修复 bug 或更新规则。

### 挑战五：规模复杂度（Scale Complexity）

当玩家数量从 2 增长到 200、2000、20000 时，网络连接的复杂度以 $O(N^2)$ 级别增长。如何构建支持万人同服的架构是下一代游戏引擎的重要课题。

## 网络协议基础

### OSI 七层模型

互联网建立在分层抽象的基础上，由 TCP/IP 的发明者（Vint Cerf 和 Robert Kahn）奠定：

| 层级 | 名称 | 说明 |
|------|------|------|
| 7 | 应用层（Application） | HTTP、FTP、游戏协议 |
| 6 | 表达层（Presentation） | 数据编码、加密 |
| 5 | 会话层（Session） | 会话管理 |
| 4 | 传输层（Transport） | **TCP / UDP** |
| 3 | 网络层（Network） | IP 路由 |
| 2 | 数据链路层（Data Link） | MAC 地址 |
| 1 | 物理层（Physical） | 电信号、光纤 |

游戏开发一般在**应用层**工作，通过 **Socket API** 与传输层交互。

### Socket 编程基础

Socket 是对传输层的抽象接口，只需提供：

- **IP 地址**：目标主机地址（IPv4 / IPv6）
- **端口号（Port）**：目标进程端口

> 注意：游戏服务器地址必须对外开放，因此需要防御 DDoS 攻击。现代网游通常有多层抗攻击机制。

**IPv4 vs IPv6**：对于面向全球市场的游戏引擎，必须同时支持 IPv6（部分国家如德国已大规模使用 IPv6）。

## TCP 协议详解

**TCP（Transmission Control Protocol）** 是互联网上最主流的传输协议，绝大多数互联网流量通过 TCP 传输。

### 核心特性

| 特性 | 描述 |
|------|------|
| **可靠连接** | 三次握手（Three-way Handshake）建立稳定连接 |
| **有序传输** | 保证数据包按发送顺序到达 |
| **流量控制** | 根据网络状况动态调整发送速率 |
| **拥塞控制** | 网络拥塞时自动降低发包速率 |

### 三次握手（Three-Way Handshake）

```
Client  --SYN-->  Server
Client  <--SYN+ACK--  Server
Client  --ACK-->  Server
（连接建立）
```

### ACK 确认机制

TCP 可靠性的核心原理是**重传机制（Retransmission Mechanism）**：

- 接收方每收到一个包，向发送方返回 **ACK（Acknowledgment）**
- 发送方只有收到 ACK 后，才继续发送后续包
- 若超时未收到 ACK，则重传该包

### 滑动窗口与拥塞控制

TCP 不是一包一包发送，而是维护一个**滑动窗口（Sliding Window）**：

$$\text{有效吞吐量} = \frac{\text{窗口大小（bytes）}}{\text{RTT（seconds）}}$$

**拥塞控制算法**（Congestion Control）：
- 初始窗口较小，传输成功则指数增大（慢启动）
- 检测到丢包，窗口减半
- 使互联网整体流量保持动态平衡

### TCP 的局限性

- **包头开销大**：包头 **20 字节**（160 位），包头比例较高
- **带宽抖动**：拥塞控制导致传输速率不稳定
- **队头阻塞（Head-of-Line Blocking）**：前面的包未确认，后续包全部阻塞
- **延迟不可控**：对实时性要求高的游戏场景不友好

> TCP 的流量控制机制是互联网能正常运行的重要保障——所有参与者自觉减速，避免网络"堵牢"。

## UDP 协议详解

**UDP（User Datagram Protocol）** 由 David P. Reed 在 TCP 之后约 10 年发明，设计哲学截然不同：极简、快速、不可靠。

### 核心特性

| 特性 | 描述 |
|------|------|
| **无连接** | 不需要握手，直接发送 |
| **无序性** | 不保证包到达顺序 |
| **无可靠性** | 不确认包是否到达 |
| **无流量控制** | 不限制发包速率 |
| **轻量包头** | 仅 **8 字节**（64 位） |

$$\text{UDP 包头开销} = 8 \text{ bytes} \approx \frac{1}{5} \times \text{TCP 包头}$$

### UDP 的优势

- **低延迟**：无握手、无 ACK 等待，响应极快（responsive）
- **支持广播/多播**：天然适合一对多分发（适合游戏服务器广播状态）
- **灵活性高**：上层协议可自定义可靠性策略

### UDP 的潜在危险

UDP 没有流量控制，可被恶意利用进行 **DDoS 放大攻击**。从开发角度，需要在 UDP 之上构建自己的可靠性保障机制。

## TCP vs UDP：游戏场景选择

实际游戏中，协议选择取决于具体场景的**延迟敏感度**：

| 游戏类型 | 延迟敏感度 | 推荐协议 | 典型案例 |
|----------|-----------|---------|----------|
| 卡牌游戏 | 低（300ms 可接受） | **TCP** | 《炉石传说》 |
| 射击/动作游戏 | 极高（<50ms） | **UDP** | 《守望先锋》 |
| MMO 游戏 | 混合 | **TCP + UDP 组合** | 《魔兽世界》 |

### 大型 MMO 的混合协议策略

```
登录认证、账号管理  →  TCP（稳定可靠）
心跳保活           →  TCP
战斗同步           →  UDP（快速响应）
聊天/邮件          →  TCP（可靠传递）
```

> 游戏引擎的核心精神：**实用主义**。没有教条，哪种协议对当前问题最有效就用哪种。

## 可靠 UDP（RUDP）

现代游戏引擎普遍选择在 UDP 之上**自定义可靠传输层**，融合 TCP 可靠性与 UDP 速度。

### RUDP 需要满足的需求

| 需求 | 来源 |
|------|------|
| 保持活跃连接（Keep-Alive） | 类 TCP |
| 一定程度的有序性 | 类 TCP |
| 高响应性、低延迟 | 类 UDP |
| 支持广播（Broadcast） | 类 UDP |
| 无头阻塞（No HOL Blocking） | 超越 TCP |

### 开源 RUDP 库推荐

- **ENet**：轻量级可靠 UDP 库，广泛用于独立游戏
- **RakNet**：功能丰富，支持多种可靠性级别
- **KCP**：高性能低延迟可靠 UDP 实现

## ARQ 重传机制

**ARQ（Automatic Repeat reQuest，自动重传请求）** 是构建 RUDP 的核心机制。

### ACK 与 NACK

- **ACK（Acknowledgment）**：告知发送方"我收到了第 N 号包"
- **NACK（Negative ACK）**：告知发送方"第 N 号包我没收到，请重传"

### 三种 ARQ 策略

#### 1. Stop-and-Wait ARQ（停等协议）

```
发送 P1 → 等待 ACK → 发送 P2 → 等待 ACK → ...
```

- 每发一包必须等待 ACK 才继续
- **缺点**：带宽严重浪费，等待 RTT（一来一回约 100-200ms）期间信道空闲

$$\text{信道利用率} = \frac{T_{发送}}{T_{发送} + 2 \times T_{传播}} \ll 1$$

#### 2. Go-Back-N ARQ（回退 N 帧）

```
发送窗口内的所有包 → 发现 P3 丢失 → 重传 P3 及之后所有包
```

- 一次发送窗口内所有包，只要收到 ACK 就滑动窗口
- 出错时回退到出错包重传所有后续包
- **推荐用于 RUDP 实现**：简单实用

#### 3. Selective Repeat ARQ（选择性重传）

```
发送窗口内的所有包 → 发现 P3 丢失（NACK）→ 仅重传 P3
```

- 利用 NACK 精确标记丢失包，只重传丢失的包
- **优点**：减少不必要重传，节约带宽
- **代价**：接收方需要乱序缓冲区，实现复杂度较高

| ARQ 类型 | 重传策略 | 带宽效率 | 实现难度 |
|----------|---------|---------|---------|
| Stop-and-Wait | 每包等 ACK | 极低 | 简单 |
| Go-Back-N | 重传出错包及后续所有包 | 中等 | 中等 |
| Selective Repeat | 仅重传出错包（NACK） | 高 | 较高 |

## 前向纠错（FEC）

**FEC（Forward Error Correction，前向纠错）** 是应对低丢包率环境的优化策略。

### 基本思路

当网络丢包率较低（如 5%~10%）时，频繁重传会引入额外延迟。FEC 通过**冗余编码**，使接收方在不请求重传的情况下恢复丢失数据。

**常用方案：XOR 冗余包**

```
发送 P1, P2, P3, P4, P5
同时发送 P_xor = P1 XOR P2 XOR P3 XOR P4 XOR P5
```

若丢失其中一个包，可通过其余包的 XOR 计算还原：

$$P_{\text{lost}} = P_1 \oplus P_2 \oplus \cdots \oplus P_{\text{xor}}$$

**更高级方案**：Reed-Solomon 纠错码，可恢复多个丢失包。

### FEC vs ARQ 权衡

| 场景 | 推荐策略 |
|------|---------|
| 丢包率 < 10%，延迟敏感 | FEC（无需重传延迟） |
| 丢包率较高，数据必须完整 | ARQ（重传保证可靠） |
| 混合场景 | FEC + ARQ 组合 |

## 关键术语表

| 术语 | 英文 | 含义 |
|------|------|------|

| **可靠 UDP** | RUDP (Reliable UDP) | 在 UDP 之上构建的可靠传输协议 |
| **自动重传** | ARQ (Auto Repeat reQuest) | 丢包时自动请求重传的机制 |
| **前向纠错** | FEC (Forward Error Correction) | 通过冗余数据恢复丢失包 |

| **端口** | Port | 标识目标进程的数字（0-65535） |

| **热更新** | Hot Update | 不停服情况下更新游戏内容 |

## 总结

本节课（第十八节上）系统介绍了**网络游戏架构的基础理论**，核心内容如下：

### 1. 网络游戏的五大挑战

- **世界一致性**：各客户端的游戏状态需保持同步
- **传输不可靠性**：丢包、延迟、断线是常态
- **安全与反作弊**：防止作弊是网络游戏的核心责任
- **设备多样性**：需支持跨平台、热更新
- **规模复杂度**：万人同服是下一代引擎的目标

### 2. 协议选择原则

```
TCP  →  可靠、有序，但延迟高、带宽抖动
UDP  →  快速、轻量，但不可靠、无序
RUDP →  两者优点结合，游戏引擎的最佳实践
```

### 3. 实践建议

- **卡牌/回合制游戏**：直接用 TCP，简单可靠
- **实时动作/射击游戏**：基于 UDP 构建 RUDP
- **大型 MMO**：TCP（登录/认证）+ UDP（战斗同步）混合架构
- **自建 RUDP**：推荐 Go-Back-N ARQ + FEC 方案，或使用 ENet/KCP 开源库

### 4. 设计哲学

> "游戏引擎是没有任何教条主义的工程科学——一切人类智慧的成果对我们来说都是工具，最终目标是构建那个'黑客帝国'式的虚拟世界。"
