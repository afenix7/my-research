## 游戏同步问题的本质

**同步（Synchronization）** 是网络游戏的核心问题：如何让分布在不同地点、通过不可靠网络相连的多台机器，对游戏世界的状态达成一致。

### 时钟同步（Clock Synchronization）

在讨论任何同步机制之前，必须解决**时钟对齐**问题。不同机器的本地时钟存在偏差（Clock Skew），需要在网络层面进行同步。

**NTP（Network Time Protocol）** 是最常见的时钟同步协议，但对游戏来说精度不够高。游戏引擎通常采用以下方法：

1. **测量 RTT**：发送时间戳包，根据往返时间估算单向延迟

$$T_{\text{delay}} \approx \frac{RTT}{2}$$

2. **客户端时间偏移**：以服务器时钟为基准，客户端维护一个偏移量 $\Delta t$

$$T_{\text{server}} = T_{\text{client}} + \Delta t$$

3. **持续校正**：定期发送心跳包，滚动平均更新 $\Delta t$

### RPC（Remote Procedure Call，远程过程调用）

**RPC** 是游戏服务器与客户端通信的基本抽象：调用远程机器上的函数，就像调用本地函数一样。

```
// 客户端调用
server.MovePlayer(playerId, newPosition, timestamp);

// 服务器执行后广播
AllClients.UpdatePlayerPosition(playerId, validatedPosition, timestamp);
```

RPC 在游戏引擎中的实现要点：
- **序列化（Serialization）**：将函数参数打包成字节流
- **反序列化（Deserialization）**：接收端还原参数
- **版本管理**：客户端与服务器的协议版本需要兼容

### 网络拓扑（Network Topology）

| 拓扑类型 | 描述 | 优点 | 缺点 | 适用场景 |
|----------|------|------|------|---------|
| **C/S（Client-Server）** | 所有客户端连接中央服务器 | 安全、权威、可扩展 | 服务器成本高，延迟略高 | 绝大多数现代网游 |
| **P2P（Peer-to-Peer）** | 客户端直接互连 | 延迟低，无服务器成本 | 难以反作弊，NAT 穿透复杂 | 早期格斗游戏、部分 RTS |
| **混合架构** | 部分数据走 P2P，关键逻辑走 C/S | 兼顾延迟与安全 | 架构复杂 | 现代大型游戏 |

**Dedicated Server（专用服务器）** 是现代 FPS/TPS 游戏的标准选择：专用服务器作为**权威节点（Authoritative Server）**，所有游戏逻辑在服务器上执行，客户端只负责渲染。

## 快照同步（Snapshot Synchronization）

**快照同步**是最直观的同步策略：服务器定期将完整的游戏世界状态（快照）广播给所有客户端。

### 工作原理

```
每帧（或每N帧）：
Server → 收集所有游戏对象的状态（位置、旋转、HP、状态...）
      → 打包成 Snapshot
      → 广播给所有客户端
Client → 接收 Snapshot
      → 更新本地世界状态
      → 渲染
```

### 快照内容

一个典型的游戏对象快照包含：

```c
struct EntitySnapshot {
    uint32_t entity_id;
    Vector3  position;
    Quat     rotation;
    Vector3  velocity;
    uint32_t health;
    uint32_t state_flags;
    uint64_t timestamp;
};
```

### 优缺点分析

**优点：**
- 实现简单，服务器是唯一权威
- 客户端逻辑极简（只需渲染服务器下发的状态）
- 天然防作弊（所有计算在服务器端）

**缺点：**
- **带宽消耗巨大**：每帧发送完整世界状态，随实体数量线性增长
- **延迟高**：客户端看到的永远是服务器的"过去时"

$$\text{带宽需求} = N_{\text{entities}} \times S_{\text{snapshot}} \times \text{TickRate}$$

### Delta Compression（增量压缩）

快照同步的核心优化：只发送**相对上一帧发生变化的部分**。

```
全量快照（Baseline）：完整游戏状态
Delta 快照：仅包含变化的字段

// 示例：只有 position 变化
Delta = {entity_id: 42, position: newPos}  // 不发 rotation/health 等
```

**优化效果**：在实体稀疏变化的情况下，带宽可降低 **70%~90%**。

**实现要点：**
- 服务器维护每个客户端的**已确认快照基线（Acknowledged Baseline）**
- 只对相对该基线发生变化的字段进行编码
- 客户端需维护基线快照以便应用 Delta

## 帧同步（Lockstep）

**帧同步（Lockstep Synchronization）** 又称**确定性同步**，其核心思想是：让所有参与者从同一初始状态出发，执行完全相同的操作序列，则最终状态必然完全一致。

### 核心原理

```
前提：游戏逻辑必须是纯确定性的（Deterministic）
        相同输入 → 相同输出，无论在哪台机器上执行

每帧：
  1. 所有玩家将本帧输入（Input）发送给所有其他玩家（或服务器）
  2. 收集到所有玩家的输入后，统一推进游戏逻辑一步
  3. 所有客户端的游戏状态在逻辑上完全一致
```

### 确定性（Determinism）的要求

帧同步成立的关键前提：

| 要求 | 说明 |
|------|------|
| **浮点数一致性** | 不同平台的浮点运算结果必须完全相同 |
| **随机数一致性** | 使用相同种子的伪随机数生成器（PRNG） |
| **物理引擎确定性** | 碰撞检测、模拟结果跨平台一致 |
| **执行顺序一致** | 所有玩家以相同顺序处理所有事件 |

> 实现真正确定性的浮点运算是帧同步最大的技术挑战，通常需要改用**定点数（Fixed-Point Arithmetic）**代替浮点数。

### Lockstep 的变体

#### 经典帧同步（P2P Lockstep）

```
玩家 A 发送输入给 B, C, D
玩家 B 发送输入给 A, C, D
...
等所有玩家的输入都收到后，统一模拟下一帧
```

**问题**：等待最慢的玩家，任何一人卡顿都拖累所有人。

#### Bucket Synchronization（桶同步）

每隔 $T$ 毫秒收集一次输入，统一处理：

$$T_{\text{bucket}} = 2 \times T_{\text{max\_latency}} + \epsilon$$

**权衡**：$T$ 越大，延迟越高；$T$ 越小，丢包风险越高。

### 帧同步的优势

- **带宽极低**：只同步玩家输入（几个字节/帧），而不是完整世界状态
- **逻辑简单**：客户端都是完整的游戏引擎，服务器仅做输入转发

### 帧同步的劣势

- **高度依赖确定性**：浮点数、随机数等必须跨平台一致
- **延迟敏感**：必须等所有玩家输入到位才能推进
- **断线处理复杂**：重连需要重新快进整个游戏历史
- **难以防作弊**：客户端有完整游戏逻辑，容易被逆向

### 典型应用

- **RTS 游戏**（《星际争霸》、《魔兽争霸 3》、《王者荣耀》）：实体数量多，帧同步带宽优势显著
- **早期格斗游戏**（GGPO 框架）

## 状态同步（State Synchronization）

**状态同步**是以服务器为权威中心，客户端发送操作/输入，服务器计算结果后将变化的游戏状态下发给客户端。

### 工作流程

```
Client → 发送操作（Move, Attack, UseSkill...）→ Server
Server → 验证操作合法性
       → 执行游戏逻辑
       → 计算状态变化
       → 向相关客户端发送状态更新（State Delta）
Client → 接收并应用状态更新
       → 渲染
```

### 与帧同步的对比

| 维度 | 帧同步（Lockstep） | 状态同步（State Sync） |
|------|-------------------|----------------------|
| **同步内容** | 玩家输入 | 游戏状态变化 |
| **服务器角色** | 输入转发（可无服务器） | 权威计算中心 |
| **带宽** | 极低（输入数据小） | 中等（状态增量） |
| **延迟** | 高（需等所有输入） | 可优化（预测） |
| **防作弊** | 难（客户端有完整逻辑） | 容易（服务器权威） |
| **确定性要求** | 严格 | 无要求 |
| **典型游戏** | RTS、MOBA | FPS、RPG、MMO |

## 航位推算（Dead Reckoning）

**Dead Reckoning（航位推算，DR）** 是一种在网络延迟下平滑预测远程实体位置的经典技术，源自航海导航。

### 问题背景

网络延迟导致客户端收到的状态更新是"过去时"的，若不做处理，远程角色会出现**跳帧（Teleport）** 或**卡顿（Stutter）** 现象。

### 基本原理

根据已知的**当前位置**、**速度**和**加速度**，预测未来位置：

$$\vec{p}(t + \Delta t) = \vec{p}(t) + \vec{v}(t) \cdot \Delta t + \frac{1}{2}\vec{a}(t) \cdot \Delta t^2$$

其中：
- $\vec{p}(t)$：当前已知位置
- $\vec{v}(t)$：当前速度
- $\vec{a}(t)$：当前加速度
- $\Delta t$：自上次同步以来的时间

### Dead Reckoning 的阶数

| 阶数 | 模型 | 适用场景 |
|------|------|---------|
| 0 阶（Zero-order） | 保持位置不变 | 静止实体 |
| 1 阶（First-order） | 匀速线性外推 | 匀速运动 |
| 2 阶（Second-order） | 匀加速外推 | 加速/变向运动 |
| 高阶 Hermite | 曲线插值 | 复杂运动轨迹 |

### 误差修正

当收到服务器的新状态更新时，本地预测位置与真实位置之间存在**误差（Dead Reckoning Error）**。

**硬纠正**（Snap）：直接跳到正确位置，产生视觉突变（不推荐）。

**软纠正**（Blend/Lerp）：在一段时间内平滑过渡：

$$\vec{p}_{\text{render}}(t) = \text{Lerp}(\vec{p}_{\text{predicted}}, \vec{p}_{\text{actual}}, \alpha)$$

其中 $\alpha$ 随时间从 0 递增到 1，过渡时间通常为 100~300ms。

### Dead Reckoning 的触发条件

不是每帧都发送更新，而是当**预测误差超过阈值**时才发送：

$$|\vec{p}_{\text{predicted}} - \vec{p}_{\text{actual}}| > \epsilon_{\text{threshold}}$$

这大幅减少了不必要的网络包，是一种**自适应发包策略**。

## 预测回滚（Prediction & Rollback）

**预测回滚（Prediction & Rollback，也称为 Client-Side Prediction + Server Reconciliation）** 是现代 FPS 游戏解决操作延迟感的核心技术。

### 问题：客户端感受到的延迟

在纯 C/S 架构下，玩家按下移动键，要等服务器确认后才能看到角色移动：

```
玩家按键 → 发包给服务器（50ms）→ 服务器计算 → 下发结果（50ms）→ 客户端渲染
总延迟 = RTT = 100ms（明显卡顿感）
```

### 客户端预测（Client-Side Prediction）

**解决方案**：客户端立即本地执行操作，不等待服务器确认：

```
玩家按键 → 立即在本地模拟移动（0ms延迟感）
         → 同时将输入发送给服务器
Server   → 验证输入，计算权威结果 → 返回
Client   → 对比服务器结果与本地预测
         → 如有偏差，进行回滚修正
```

### 服务器对账（Server Reconciliation）

客户端需要维护一个**未确认输入队列**：

```c
struct InputRecord {
    uint32_t sequence_number;  // 输入序列号
    Input    input;            // 输入内容
    State    predicted_state;  // 预测的状态
};

queue<InputRecord> unacknowledged_inputs;
```

当收到服务器的权威状态时：
1. 找到服务器确认的最后输入序列号
2. 将游戏状态回滚到该序列号对应的时刻
3. 重新应用所有未确认的输入（Replay）
4. 若最终状态与本地预测一致，则无感知；若有偏差，则平滑纠正

### Rollback Netcode（回滚网络编码）

格斗游戏中的 **GGPO（Good Game Peace Out）** 框架将 Rollback 发挥到极致：

```
假设 Player A 和 Player B 通过 P2P 连接

每帧：
1. 用对方的最新已知输入（或预测输入）立即模拟本帧
2. 当收到对方的真实输入时：
   - 若与预测一致：继续
   - 若不一致：
     a. 回滚到分叉帧
     b. 用真实输入重新模拟到当前帧
     c. 若回滚帧数少（≤ 8帧），玩家几乎无感知
```

**Rollback 的关键限制**：回滚帧数不能太多，否则产生明显的时间倒流感：

$$N_{\text{rollback}} \leq \frac{RTT / 2}{T_{\text{frame}}}$$

例如：RTT = 100ms，帧率 = 60fps → 最多回滚 3 帧

## 延迟补偿技术

### 延迟补偿（Lag Compensation）

FPS 游戏中的经典问题：玩家看到对手在位置 A，开枪，但服务器收到开枪请求时对手已经移动到位置 B。

**解决方案**：服务器维护**历史状态缓冲区（History Buffer）**，在处理射击请求时，将世界回退到玩家开枪时刻的状态来做命中判断：

```
Server 接收到开枪请求（带时间戳 T）：
1. 查询 T 时刻的世界状态快照
2. 在该历史状态下执行命中检测（Hit Detection）
3. 应用伤害
4. 广播结果
```

$$T_{\text{rewind}} = T_{\text{server\_receive}} - T_{\text{client\_send}} \approx \frac{RTT}{2}$$

**权衡**：这意味着被射到的玩家可能感觉"已经躲开了"却还是中弹，但对射击方来说体验更公平。

### 插值（Interpolation）vs 外推（Extrapolation）

**插值（Interpolation）**：在已知的两个状态之间平滑过渡（适合呈现过去状态）：

```
状态 A (t=100ms)  →  插值  →  状态 B (t=200ms)
```

$$\vec{p}(t) = \text{Lerp}(\vec{p}_A, \vec{p}_B, \frac{t - t_A}{t_B - t_A})$$

**外推（Extrapolation）**：基于最新已知状态预测未来（即 Dead Reckoning）。

**插值缓冲区（Interpolation Buffer）**：客户端故意保留 100~200ms 的延迟，确保始终有足够多的状态帧可供插值，消除网络抖动（Jitter）的视觉影响。

## 网络优化技术

### Delta Compression（增量压缩）

（见[快照同步章节](#delta-compression增量压缩)）

核心原理：只发送相对基线（Baseline）发生变化的字段。

实现时需要注意：
- **位压缩（Bit Packing）**：对小范围值用少量比特编码（如方向角用 16 位代替 32 位浮点数）
- **字典压缩**：对重复出现的值进行字典映射

### Interest Management（兴趣管理）

在 MMO 等大世界游戏中，不可能把所有实体的状态同步给每个玩家。**兴趣管理（Interest Management，IM）** 只同步每个玩家"感兴趣"的实体。

#### Area of Interest（AoI，兴趣区域）

```
玩家只接收其周围 R 半径内的实体状态更新
超出范围的实体不同步
```

$$N_{\text{sync}} \approx \rho \cdot \pi R^2 \ll N_{\text{total}}$$

其中 $\rho$ 是实体密度，$R$ 是兴趣半径。

#### 动态 AoI 策略

- 根据实体的**重要性**动态调整 AoI 半径
- Boss、重要 NPC 的同步范围比普通玩家大
- 视线（Line of Sight）裁剪：看不见的实体降低更新频率

### 数据压缩与编码

| 技术 | 应用场景 | 收益 |
|------|---------|------|
| **位压缩（Bit Packing）** | 枚举值、有限范围整数 | 减少 30~50% 包大小 |
| **Delta 编码** | 连续变化的数值 | 减少 60~90% 包大小 |
| **哈夫曼编码** | 频繁出现的值 | 减少 20~40% 包大小 |
| **浮点量化** | 位置、旋转等 | 减少 50~75% 包大小 |

### 优先级与频率控制

不同实体/数据的更新频率不同：

| 数据类型 | 更新频率 | 说明 |
|----------|---------|------|
| 玩家位置 | 20~60 Hz | 高频，实时感要求高 |
| NPC 位置 | 5~10 Hz | 中频 |
| 环境状态 | 1~5 Hz | 低频 |
| 聊天消息 | 事件触发 | 按需发送 |

## MMO 服务器架构

### 区域分割（Zone/Region Architecture）

大世界 MMO 通常按地图区域分割，每个区域由独立的**Zone Server**负责：

```
┌─────────────────────────────────────────┐
│              Login Server               │
└─────────────────────┬───────────────────┘
                      │
        ┌─────────────┼─────────────┐
        ↓             ↓             ↓
   Zone Server 1  Zone Server 2  Zone Server 3
   （城市区域）    （野外区域）    （副本区域）
```

**跨区域同步**：玩家跨区时，Zone Server 之间需要进行**无缝切换（Seamless Transfer）**。

### Shared World Architecture（共享世界架构）

更现代的架构是让同一区域的实体数据存储在共享数据层，多个 Zone Server 可以同时读写：

```
Zone Server A ─┐
               ├─→ Shared State Store（Redis/自定义）
Zone Server B ─┘
```

### 微服务化

现代网游服务器通常拆分为多个独立微服务：

| 服务 | 职责 |
|------|------|
| **Gateway Server** | 负载均衡、连接管理 |
| **Login/Auth Server** | 认证、账号管理 |
| **Game Logic Server** | 游戏逻辑计算 |
| **Chat Server** | 聊天系统 |
| **Matchmaking Server** | 玩家匹配 |
| **Database Proxy** | 数据库访问抽象 |

## 关键术语表

| 术语 | 英文 | 含义 |
|------|------|------|
| **快照同步** | Snapshot Synchronization | 定期广播完整世界状态的同步方式 |
| **增量压缩** | Delta Compression | 只传输相对基线变化部分的压缩策略 |
| **帧同步** | Lockstep Synchronization | 所有玩家共享相同输入序列保证一致性 |
| **状态同步** | State Synchronization | 服务器权威计算后下发状态变化 |
| **航位推算** | Dead Reckoning | 基于速度/加速度预测未来位置 |
| **客户端预测** | Client-Side Prediction | 客户端本地立即执行操作不等服务器 |
| **服务器对账** | Server Reconciliation | 用服务器权威结果修正本地预测 |
| **回滚网络** | Rollback Netcode | 通过状态回滚+重放消除输入延迟 |
| **延迟补偿** | Lag Compensation | 服务器回退历史状态处理命中判断 |
| **插值** | Interpolation | 在已知状态间平滑过渡 |
| **外推** | Extrapolation | 基于当前状态预测未来 |
| **兴趣管理** | Interest Management | 只同步玩家感兴趣的实体状态 |
| **兴趣区域** | AoI (Area of Interest) | 实体接收状态更新的范围 |
| **确定性** | Determinism | 相同输入必然产生相同输出 |
| **定点数** | Fixed-Point Arithmetic | 用整数模拟小数，保证跨平台一致 |
| **权威服务器** | Authoritative Server | 游戏逻辑的最终权威计算节点 |
| **抖动** | Jitter | 网络延迟的不规则波动 |
| **插值缓冲** | Interpolation Buffer | 故意引入延迟以平滑抖动 |
| **无缝切换** | Seamless Transfer | 玩家跨 Zone 时无感知迁移 |
| **RPC** | Remote Procedure Call | 远程过程调用 |
| **心跳包** | Heartbeat Packet | 保持连接活跃的定期空包 |

## 总结

本节课（第十八节下）深入讲解了网络游戏的**同步机制与优化技术**，核心内容如下：

### 1. 三大同步范式对比

```
快照同步（Snapshot）
  → 完整状态广播，实现简单，带宽消耗高
  → 适合实体少、服务器算力足够的场景

帧同步（Lockstep）
  → 同步输入，带宽极低，需严格确定性
  → 适合 RTS/MOBA，客户端有完整逻辑

状态同步（State Sync）
  → 服务器权威，安全性高，需要预测补偿
  → 适合 FPS/MMO，现代游戏主流选择
```

### 2. 延迟消除技术链

```
网络延迟
  ↓ Client-Side Prediction（客户端预测）
本地操作即时响应
  ↓ Server Reconciliation（服务器对账）
修正预测偏差
  ↓ Dead Reckoning（航位推算）
远程实体平滑移动
  ↓ Lag Compensation（延迟补偿）
公平的命中判断
  ↓ Interpolation Buffer（插值缓冲）
消除网络抖动的视觉影响
```

### 3. 带宽优化技术链

```
原始状态数据
  ↓ Delta Compression（增量压缩）
  ↓ Interest Management（兴趣管理）
  ↓ Bit Packing（位压缩）
  ↓ 频率控制（重要性排序）
最终网络传输量大幅降低
```

### 4. 设计原则

| 原则 | 说明 |
|------|------|
| **权威服务器** | 关键游戏逻辑必须在服务器执行，防作弊 |
| **乐观执行** | 客户端先执行再对账，消除延迟感 |
| **渐进降级** | 网络差时降低更新频率，而非卡死 |
| **数据最小化** | 只传输必要数据，时刻思考带宽 |
| **实用主义** | 没有万能方案，根据游戏类型选择合适策略 |

### 5. 技术选型建议

| 游戏类型 | 同步方案 | 预测方案 | 带宽优化 |
|----------|---------|---------|---------|
| **卡牌/回合制** | 状态同步 | 无需预测 | 无需优化 |
| **MOBA/RTS** | 帧同步 | Rollback（GGPO） | 仅同步输入 |
| **FPS/TPS** | 状态同步 | Client Prediction + Lag Comp | Delta + AoI |
| **MMO 大世界** | 状态同步 + 快照基线 | Dead Reckoning | Delta + AoI + 频率控制 |
| **格斗游戏** | 帧同步 | GGPO Rollback | 仅同步输入 |
