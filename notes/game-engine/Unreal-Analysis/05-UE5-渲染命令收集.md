## 1. 概述

在 UE5 渲染架构中，底层 RHI（Render Hardware Interface）和 DX12 层面已经有了渲染命令的封装（见系列第 4 集）。而在**上层逻辑**（引擎渲染逻辑层），同样需要一套机制，将整个场景中所有的网格、材质、着色器等信息**收集**起来，形成统一的数据结构，再批量转换为 RHI 层面的命令调用。

本集的核心问题：
- 上层逻辑如何表示"一次绘制所需的所有信息"？
- 这些信息如何从场景中被**收集**？
- 最终如何**转换**为 RHI 指令？

答案的中心是两个关键数据结构：**`FMeshDrawCommand`** 和 **`FMeshBatch`**，以及连接它们的 **`BuildMeshDrawCommand`** 函数。

---

## 2. 核心数据结构：FMeshDrawCommand

`FMeshDrawCommand` 是 UE5 上层渲染逻辑中**一次绘制调用所需全部信息**的封装。它代表了一条完整的渲染命令，包含从 Pipeline State 到顶点数据的所有内容。

### 2.1 成员变量

| 成员变量 | 类型 | 说明 |
|----------|------|------|
| Shaders（含 VS/PS/MS 等） | `FMeshDrawShaderBindings` | 着色器及其绑定的 Uniform Buffer、Texture、SRV、Sampler 等 |
| VertexStreams | `TArray<FVertexInputStream>` | 顶点输入流数组，每个元素包含 RHI Buffer + 偏移 |
| IndexBuffer | `FRHIBuffer*` | 索引缓冲区 |
| PipelineId / PSO | `FGraphicsPipelineState` | Pipeline State Object，包含光栅化状态、深度/模板测试、混合状态等 |
| PrimitiveType | `EPrimitiveType` | 三角形类型（TriList、TriStrip 等） |
| 其他参数 | 各类标志位 | LOD、动态/静态标记等 |

典型代码结构（简化）：

```cpp
class FMeshDrawCommand
{
public:
    // 着色器绑定：包含 VS/PS/MS 等所有着色器及其资源绑定
    FMeshDrawShaderBindings ShaderBindings;

    // 顶点输入流
    TArray<FVertexInputStream, TInlineAllocator<4>> VertexStreams;

    // 索引缓冲区
    FRHIBuffer* IndexBuffer = nullptr;

    // 管线状态（内部存的是一个 ID，通过 Get 接口取出实际 PSO）
    FGraphicsPipelineStateId CachedPipelineId;

    // 绘制参数
    uint32 FirstIndex;
    uint32 NumPrimitives;
    uint32 NumInstances;

    // 提交绘制命令
    static void SubmitDraw(
        const FMeshDrawCommand& MeshDrawCommand,
        FRHICommandList& RHICmdList,
        ...
    );
};
```

### 2.2 SubmitDraw：提交到 RHI

`SubmitDraw` 是一个**静态函数**，负责将 `FMeshDrawCommand` 里收集到的所有信息一次性转换为 RHI 调用。它分为两个逻辑阶段：

**阶段一：Begin（设置所有状态）**

```
1. 从 CachedPipelineId 取出 FGraphicsPipelineState
2. 调用 RHICmdList.SetGraphicsPipelineState(PSO)
   → 内部调用平台相关 Context，真正设置 D3D12/Vulkan 管线
3. 设置模板引用值（Stencil Ref）
4. 遍历 VertexStreams，逐一调用 RHICmdList.SetStreamSource(...)
5. 绑定着色器资源：
   - Vertex Shader → SetShaderUniformBuffer / SetShaderTexture / SetShaderSRV
   - Pixel Shader  → 同上
   - Mesh/Amplification Shader → 同上
```

**阶段二：End（发起绘制）**

```cpp
// 根据是否有 IndexBuffer 分支：
if (IndexBuffer)
{
    RHICmdList.DrawIndexedPrimitive(
        IndexBuffer,
        FirstIndex, NumPrimitives, NumInstances, ...
    );
}
else
{
    RHICmdList.DrawPrimitive(...);
}
```

> 说明：Begin/End 的分离设计允许在某些情况下只做状态设置而不实际发起 Draw，便于批量优化。

## 3. 中间层数据结构：FMeshBatch

在场景收集阶段，并不是直接生成 `FMeshDrawCommand`，而是先生成 **`FMeshBatch`**，再由 `BuildMeshDrawCommand` 转换。

### 3.1 FMeshBatch

```cpp
struct FMeshBatch
{
    // 元素数组（多数情况下只有 1 个元素）
    TArray<FMeshBatchElement, TInlineAllocator<1>> Elements;

    // 共享的顶点工厂（所有 Elements 共用）
    const FVertexFactory* VertexFactory;

    // 共享的材质代理（所有 Elements 共用）
    const FMaterialRenderProxy* MaterialRenderProxy;

    // 图元类型
    uint8 Type;         // EPrimitiveType

    // 其他标志
    bool bWireframe;
    bool bUseForMaterial;
    // ...
};
```

### 3.2 FMeshBatchElement

`FMeshBatchElement` 是 `FMeshBatch` 内部数组的单个元素，代表一个具体的绘制单元：

```cpp
struct FMeshBatchElement
{
    // 索引缓冲区
    const FRawIndexBuffer* IndexBuffer;

    // 绘制范围
    uint32 FirstIndex;
    uint32 NumPrimitives;
    uint32 NumInstances;

    // 顶点缓冲区（可选，部分平台需要）
    const FVertexBufferRHIRef* VertexBuffer;

    // 可见性 mask
    uint64 VisibilityMask;

    // Primitive Uniform Buffer（变换矩阵等）
    FUniformBufferRHIRef PrimitiveUniformBuffer;
    // ...
};
```

### 3.3 为什么需要 FMeshBatch

历史原因（约 UE 4.20 之前），引擎只有 `FMeshBatch` 这个中间结构，没有 `FMeshDrawCommand`。后来引入 `FMeshDrawCommand` 作为最终的命令封装，但为了兼容性和职责分离，保留了两层结构：

| 结构 | 职责 | 来源 |
|------|------|------|
| `FMeshBatch` | 表达"这个模型用什么材质、顶点工厂、索引范围" | `FPrimitiveSceneProxy::GetDynamicMeshElements` 生成 |
| `FMeshDrawCommand` | 表达"如何调用 RHI 完成这次绘制" | `BuildMeshDrawCommand` 根据 FMeshBatch 生成 |

## 4. BuildMeshDrawCommand：从 FMeshBatch 到 FMeshDrawCommand

`BuildMeshDrawCommand` 是一个**模板函数**，负责将 `FMeshBatch` 转换为一个或多个 `FMeshDrawCommand`。

```cpp
template <typename PassShadersType, typename ShaderElementDataType>
void BuildMeshDrawCommand(
    FRHICommandList& RHICmdList,
    const FMeshBatch& MeshBatch,
    const FPrimitiveSceneProxy* PrimitiveSceneProxy,
    const FMaterialRenderProxy& MaterialRenderProxy,
    const FMaterial& MaterialResource,
    const FMeshPassProcessorRenderState& DrawRenderState,
    PassShadersType PassShaders,
    EMeshPassFeatures MeshPassFeatures,
    const ShaderElementDataType& ShaderElementData,
    FMeshPassDrawListContext* DrawListContext
);
```

**执行逻辑**（简化）：

```
1. 收集公共数据（shared data）：
   - 从 MeshBatch.VertexFactory 获取 Pipeline State 初始化器
   - 收集公共的顶点工厂绑定（Vertex Factory Shader Bindings）
   - 将公共数据存入 SharedMeshDrawCommand（作为模板）

2. for (int32 BatchElementIndex = 0;
         BatchElementIndex < MeshBatch.Elements.Num();
         BatchElementIndex++)
   {
       // 以 SharedMeshDrawCommand 为基础，分配一个新的 FMeshDrawCommand
       FMeshDrawCommand& NewCommand = DrawListContext->AddCommand(SharedMeshDrawCommand);

       // 收集每个 Element 特有的数据：
       // - Vertex Shader 绑定
       // - Pixel Shader 绑定
       // - Primitive Uniform Buffer
       // - 三角形绘制范围（FirstIndex、NumPrimitives）
       // - 可见性相关参数

       // Finalize：完成命令创建
       DrawListContext->FinalizeCommand(..., NewCommand);
   }
```

> 关键点：`FMeshBatch` 中公共的 `VertexFactory` 和 `MaterialRenderProxy` 只处理一次，然后复制给每个 `FMeshDrawCommand`，避免重复计算。

## 5. FMeshElementCollector：收集器

`FMeshElementCollector` 是 `FMeshBatch` 的容器/收集器，由 `FSceneRenderer`（场景渲染器）持有，在 `GatherDynamicMeshElements` 阶段被填充。

**核心成员（简化）**：

```cpp
class FMeshElementCollector
{
    // 各个 View 对应的 MeshBatch 数组（数组的数组）
    TArray<TArray<FMeshBatch*>> MeshBatches;

    // 简单几何体（调试用直线、点等）
    TArray<FSimpleElementEntry> SimpleElements;

    // 材质相关缓存
    TArray<const FMaterialRenderProxy*> MaterialProxies;

    // 每个 MeshBatch 对应的数量/LOD 等信息
    TArray<int32> NumMeshBatchElementsPerView;

public:
    // 核心接口：分配并返回一个 FMeshBatch 引用
    FMeshBatch& AllocateMesh();

    // 注册已填充好的 MeshBatch
    void AddMesh(int32 ViewIndex, int32 MeshBatchIndex);
};
```

**调试用简单几何体**：收集器中还包含 `SimpleElements`，用于存放 Debug 时的直线、点、面等简单图元，不走 `FMeshBatch` 流程。

## 6. 完整收集流程

### 6.1 GatherDynamicMeshElements

`FSceneRenderer::GatherDynamicMeshElements` 是动态 Mesh 收集的入口：

```cpp
void FSceneRenderer::GatherDynamicMeshElements(
    TArray<FViewInfo>& Views,
    const FScene* Scene,
    const FSceneViewFamily& ViewFamily,
    const FPrimitiveViewMasks& HasDynamicMeshElementsMasks,
    const FPrimitiveViewMasks& HasDynamicEditorMeshElementsMasks,
    FMeshElementCollector& Collector
)
{
    // 1. 清空 Collector
    Collector.ClearMeshBatches();

    for (int32 PrimitiveIndex : DynamicPrimitives)
    {
        FPrimitiveSceneProxy* Proxy = Scene->Primitives[PrimitiveIndex]->Proxy;

        // 2. 可见性剔除：不可见的 Primitive 不收集
        if (!IsVisible(Proxy, Views)) continue;

        // 3. 设置 Primitive 信息（变换等）
        Collector.SetPrimitive(Proxy, BatchHitProxyId);

        // 4. 调用 Proxy 的虚函数填充 MeshBatch
        Proxy->GetDynamicMeshElements(Views, ViewFamily, HasDynamicMeshElementsMasks, Collector);
    }
}
```

### 6.2 FPrimitiveSceneProxy

`FPrimitiveSceneProxy` 是场景图元在渲染线程的代理，负责实际生成 `FMeshBatch`：

```cpp
// 虚函数，由各类图元子类实现
virtual void GetDynamicMeshElements(
    const TArray<const FSceneView*>& Views,
    const FSceneViewFamily& ViewFamily,
    uint32 VisibilityMap,
    FMeshElementCollector& Collector
) const
{
    // 遍历所有 View
    for (int32 ViewIndex = 0; ViewIndex < Views.Num(); ViewIndex++)
    {
        if (VisibilityMap & (1 << ViewIndex))
        {
            // 向 Collector 分配一个 FMeshBatch
            FMeshBatch& MeshBatch = Collector.AllocateMesh();

            // 填充 FMeshBatchElement（通常只有 [0] 一个元素）
            FMeshBatchElement& BatchElement = MeshBatch.Elements[0];
            BatchElement.IndexBuffer    = &IndexBuffer;
            BatchElement.FirstIndex     = 0;
            BatchElement.NumPrimitives  = NumTriangles;
            BatchElement.NumInstances   = 1;
            BatchElement.PrimitiveUniformBuffer = GetUniformBuffer();

            // 填充公共信息
            MeshBatch.VertexFactory          = &VertexFactory;
            MeshBatch.MaterialRenderProxy    = Material->GetRenderProxy();
            MeshBatch.Type                   = PT_TriangleList;

            // 注册到 Collector
            Collector.AddMesh(ViewIndex, MeshBatchIndex);
        }
    }
}
```

### 6.3 Pass 子类实现

场景渲染器有很多子 Pass（如 Base Pass、Shadow Pass、Depth Pass 等），每个 Pass 继承自 `FMeshPassProcessor`，通过不同的 Shader 策略（Policy）实现 `FMeshPassProcessor::AddMeshBatch`：

```cpp
// Base Pass 示例（简化）
class FBasePassMeshProcessor : public FMeshPassProcessor
{
public:
    virtual void AddMeshBatch(
        const FMeshBatch& MeshBatch,
        uint64 BatchElementMask,
        const FPrimitiveSceneProxy* PrimitiveSceneProxy,
        int32 StaticMeshId = -1
    ) override
    {
        // 选择合适的 Shader
        TShaderMapRef<TBasePassVS<...>> VertexShader(ShaderMap);
        TShaderMapRef<TBasePassPS<...>> PixelShader(ShaderMap);

        // 调用 BuildMeshDrawCommand 生成 FMeshDrawCommand
        BuildMeshDrawCommand(
            RHICmdList,
            MeshBatch,
            PrimitiveSceneProxy,
            *MeshBatch.MaterialRenderProxy,
            *Material,
            DrawRenderState,
            PassShaders,
            MeshPassFeatures,
            ShaderElementData,
            DrawListContext
        );
    }
};
```

## 7. 与 RHI 层的关系

本集内容处于整个渲染架构的中间层，向上承接场景逻辑，向下对接 RHI：

```
场景游戏逻辑层
  ↓ Actor / Component / UPrimitiveComponent
  ↓ FPrimitiveSceneProxy（渲染线程代理）
  ↓ GetDynamicMeshElements()
  ↓ FMeshElementCollector 收集 FMeshBatch
  ↓ BuildMeshDrawCommand()
  ↓ FMeshDrawCommand（上层命令封装）
  ↓ SubmitDraw()
  ↓ FRHICommandList（RHI 命令列表）
  ↓ SetGraphicsPipelineState / SetStreamSource / DrawIndexedPrimitive
  ↓ D3D12 / Vulkan / Metal 平台实现
```

**与 DX12 层缓存机制的对应**：上一集介绍过 DX12 层面在调用 `SetXxx` 时会先缓存到 State 中，真正在 `FlushState` 时才批量提交。`FMeshDrawCommand` 在上层做的事情与此类似：先把所有状态收集到一个对象里，再通过 `SubmitDraw` 一次性转换为 RHI 调用。

### 9.2 FMeshBatch 内部结构

```
FMeshBatch
├── VertexFactory*          ← 共享
├── MaterialRenderProxy*    ← 共享
├── Type (PT_TriangleList)  ← 共享
└── Elements[]
    ├── [0] FMeshBatchElement
    │    ├── IndexBuffer*
    │    ├── FirstIndex
    │    ├── NumPrimitives
    │    ├── NumInstances
    │    └── PrimitiveUniformBuffer
    ├── [1] FMeshBatchElement  （多 LOD 或实例化时）
    └── ...
```

### 9.3 BuildMeshDrawCommand 内部逻辑

```
BuildMeshDrawCommand(MeshBatch, ...)
        │
        ├── [共享阶段]
        │     SharedCmd.PipelineId  = GetPSO(VertexFactory, Material, ...)
        │     SharedCmd.VertexStreams = VertexFactory.GetStreams()
        │
        └── for (Element : MeshBatch.Elements)
              │
              ├── NewCmd = DrawListContext.AddCommand(SharedCmd)
              ├── NewCmd.ShaderBindings.VS = VS.GetBindings(Element)
              ├── NewCmd.ShaderBindings.PS = PS.GetBindings(Element)
              ├── NewCmd.FirstIndex  = Element.FirstIndex
              ├── NewCmd.NumPrimitives = Element.NumPrimitives
              └── DrawListContext.FinalizeCommand(NewCmd)
```

## 10. 历史背景与设计演化

| 版本 | 渲染命令机制 |
|------|-------------|
| 早期（< 4.20 左右） | 只有 `FMeshBatch`，收集完直接调 RHI |
| UE 4.22+ | 引入 `FMeshDrawCommand`，实现命令缓存与重用（静态 Mesh 的 Draw Command 可复用） |
| UE 5.x | 延续双层结构，增加 Nanite、RDG 等新机制，但 `FMeshBatch → FMeshDrawCommand` 流程不变 |

引入 `FMeshDrawCommand` 的主要收益：
- **静态 Mesh 命令复用**：静态场景对象的 Draw Command 可在每帧渲染时直接复用，无需重新 Build。
- **并行构建**：多线程并行生成 `FMeshDrawCommand`，提升构建效率。
- **与 RDG 解耦**：命令的构建（Build）和提交（Submit）可分开进行，便于 RDG 的 Pass 依赖分析和资源屏障插入。

## 11. 总结

本集建立了 UE5 渲染命令收集的整体框架，核心流程如下：

```
场景图元（Actor/Component）
    → FPrimitiveSceneProxy（渲染线程代理）
    → GetDynamicMeshElements()
    → FMeshElementCollector 收集 FMeshBatch
    → BuildMeshDrawCommand()
    → FMeshDrawCommand（完整绘制命令）
    → SubmitDraw()
    → RHI 命令（SetPSO / SetStreamSource / DrawIndexed）
    → D3D12/Vulkan 平台
```

**关键认知**：
1. `FMeshDrawCommand` 是上层对"一次 Draw Call"的完整封装，包含 PSO、Vertex Streams、Shader Bindings、Draw 参数等全部信息。
2. `FMeshBatch` 是更靠近场景描述的中间层，由各 `FPrimitiveSceneProxy` 子类生成，表达"这个模型用哪些顶点、哪个材质"。
3. `BuildMeshDrawCommand` 负责从 `FMeshBatch` 生成 `FMeshDrawCommand`，其中利用 shared 结构避免重复处理公共数据。
4. `FMeshElementCollector` 是收集器，内部是 per-View 的 `FMeshBatch` 数组。
5. 不同 Pass（Base Pass、Shadow Pass 等）各自继承 `FMeshPassProcessor` 并实现 `AddMeshBatch`，选择对应 Shader 后调用 `BuildMeshDrawCommand`。

下一集将继续深入各个具体 Pass（如 Base Pass）的实现细节。
