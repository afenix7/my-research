## 概述

本集分析 UE5.1 引擎中**渲染命令（Render Command）的封装体系**，分两个层面讲解：

| 层面 | 内容 |
|------|------|
| **RHI 抽象层** | 命令基类、命令链表、命令创建与执行的通用框架 |
| **D3D12 后端** | 三条命令队列、状态缓存、Payload 提交、专属渲染线程 |

核心设计思路：**命令的创建与命令的执行分离**。上层（游戏线程 / RHI 渲染线程）把命令写入链表，底层（D3D12 线程）在合适的时机批量提交给 GPU。

---

## RHI 层渲染命令结构

### FRHICommandBase：命令基类

所有 RHI 命令都继承自 `FRHICommandBase`，其核心结构极为精简：

```cpp
struct FRHICommandBase
{
    FRHICommandBase* Next = nullptr;  // 链表指针，指向下一条命令

    // 执行并销毁该命令（纯虚，由子类实现）
    virtual void ExecuteAndDestruct(FRHICommandListBase& CmdList,
                                    FRHICommandListExecuteContext& Context) = 0;
};
```

- `Next` 指针使所有命令天然构成**单向链表**。
- `ExecuteAndDestruct()` 是虚函数，在真正执行时调用子类实现，执行完后就地析构并释放内存。

---

### TRHILambdaCommand：Lambda 命令模板

为了方便使用 Lambda 直接创建命令，RHI 提供了一个辅助模板类：

```cpp
template <typename Lambda>
struct TRHILambdaCommand final : public FRHICommandBase
{
    Lambda StoredLambda;

    TRHILambdaCommand(Lambda&& InLambda)
        : StoredLambda(MoveTemp(InLambda)) {}

    void ExecuteAndDestruct(FRHICommandListBase& CmdList,
                            FRHICommandListExecuteContext& Context) override
    {
        StoredLambda(CmdList);
        StoredLambda.~Lambda();  // 执行后立刻销毁 Lambda，释放捕获资源
    }
};
```

**作用**：无需为每一种操作都手写子类，直接把 Lambda 包裹成命令对象插入链表。这正是引擎外部常见的 `ENQUEUE_RENDER_COMMAND` 宏的底层机制。

---

### FRHICommandList：命令链表

`FRHICommandList`（继承自 `FRHICommandListBase`）负责维护命令的链表结构：

```cpp
class FRHICommandListBase
{
protected:
    FRHICommandBase* Root = nullptr;       // 链表头：第一条命令
    FRHICommandBase** Tail = &Root;        // 指向最后一条命令的 Next 字段的指针

    uint32 NumCommands = 0;                // 命令计数
    // ...
};
```

**关键设计：`Tail` 是二级指针**

`Tail` 指向"最后一条命令的 `Next` 字段"，而不是指向最后一条命令本身。这样在追加新命令时：

```
追加前：Root → Cmd1 → Cmd2 → nullptr
                              ↑
                           Tail 指向这里（Cmd2.Next 的地址）

追加后：Root → Cmd1 → Cmd2 → Cmd3 → nullptr
                                      ↑
                                   Tail 更新到 Cmd3.Next 的地址
```

O(1) 时间追加，无需遍历整个链表。

---

### AllocateCommand：命令分配与自动入队

```cpp
template <typename TCmd>
TCmd* AllocateCommand()
{
    TCmd* Cmd = new (AllocMemory(sizeof(TCmd))) TCmd(); // Placement new

    *Tail = Cmd;           // 把新命令挂到链表尾部
    Tail = &Cmd->Next;     // 更新 Tail 到新命令的 Next 字段

    NumCommands++;
    return Cmd;
}
```

**关键含义**：`AllocateCommand()` 一旦调用，命令就**自动进入链表**。外部代码只需关注"创建命令"，不需要手动调用"加入队列"的函数。这在阅读源码时可能让人困惑——"为什么只见到 new 命令，却不见入队操作？"答案正在于此。

## 命令链表的遍历与执行

`FRHICommandListBase` 提供了 `Execute()` 方法，通过迭代器遍历并执行所有命令：

```cpp
void FRHICommandListBase::Execute(FRHICommandListExecuteContext& Context)
{
    FRHICommandBase* CmdPtr = Root;
    while (CmdPtr)
    {
        FRHICommandBase* Next = CmdPtr->Next; // 先保存 Next，因为 ExecuteAndDestruct 会析构对象
        CmdPtr->ExecuteAndDestruct(*this, Context);
        CmdPtr = Next;
    }
    Root = nullptr;
    Tail = &Root;
    NumCommands = 0;
}
```

执行流程：
1. 从 `Root` 出发，沿 `Next` 指针逐条遍历
2. 调用每条命令的虚函数 `ExecuteAndDestruct()`，执行完毕后自动析构
3. 遍历完成，重置链表

## RHI 命令类型体系

### 具体命令子类举例

以设置 Shader 参数的命令为例：

```cpp
// 设置 Shader 参数（以 SetShaderParameter 为例）
struct FRHICommandSetShaderParameter : public FRHICommand<FRHICommandSetShaderParameter>
{
    FRHIGraphicsShader* Shader;
    uint32 BufferIndex;
    uint32 BaseIndex;
    uint32 NumBytes;
    const void* NewValue;

    void Execute(FRHICommandListBase& CmdList)
    {
        // 通过 IRHICommandContext 调用平台实现
        Context->RHISetShaderParameter(Shader, BufferIndex, BaseIndex, NumBytes, NewValue);
    }
};
```

引擎中类似的命令子类有数百个，涵盖：

| 命令类别 | 示例 |
|----------|------|
| Shader 参数 | `SetShaderParameter`, `SetShaderUniformBuffer` |
| 纹理/资源绑定 | `SetShaderTexture`, `SetShaderSRV`, `SetShaderUAV` |
| Sampler | `SetShaderSampler` |
| 顶点/索引缓冲 | `SetStreamSource`, `SetIndexBuffer` |
| Uniform Buffer | `BuildLocalUniformBuffer` |
| 光栅化状态 | `SetViewport`, `SetScissorRect`, `SetBlendFactor` |
| Draw Call | `DrawPrimitive`, `DrawIndexedPrimitive` |

---

### IRHIComputeContext 与 IRHICommandContext

RHI 上下文有两层继承：

```
IRHIComputeContext          <- 计算着色器相关操作（SetComputeShader、DispatchComputeShader 等）
        ↑
IRHICommandContext           <- 图形渲染操作（Draw、SetViewport、SetRenderTargets 等）
        ↑
FD3D12CommandContext         <- D3D12 平台具体实现（虚函数的落地）
```

- `IRHIComputeContext`：定义 Compute 相关的纯虚接口（SetUAV、SetCBV、SetSRV、Dispatch…）
- `IRHICommandContext`：继承 Compute Context，追加所有图形接口（Draw、设置管线状态等）
- 这些接口在子类里通过 `FD3D12StateCache` 做状态缓存，而不是直接调 D3D12 API

---

### FRHICommandListImmediate：立即执行模式

```cpp
class FRHICommandListImmediate : public FRHICommandList
{
    // 立即执行：命令入队后不等批量提交，立刻在当前线程执行
    // 内部仍通过 AllocateCommand / Execute 走相同路径
};
```

立即模式适用于需要同步等待结果的操作（如读回 GPU 数据），普通渲染 Pass 则使用延迟的 `FRHICommandList`。

---

### FDynamicRHI 与 Context 的区别

| 接口 | 所属类 | 特点 |
|------|--------|------|
| `CreateTexture2D()` / `CreateBuffer()` | `FDynamicRHI` | 与当前绘制上下文**无关**，直接调用 Device 级 API |
| `SetShaderParameter()` / `Draw()` | `IRHICommandContext` | 需要当前的命令列表 / 管线上下文才能执行 |

D3D12 中所有绘制操作都依赖于 `ID3D12GraphicsCommandList`，而不同的绘制命令可能在不同的 CommandList 上发起，因此必须通过 Context 区分当前状态。

## D3D12 后端的实现

### FD3D12Device：设备与三条命令队列

```cpp
class FD3D12Device
{
    // 每个 D3D12 Device 对应三条命令队列
    FD3D12Queue Queues[ED3D12QueueType::Count];
    // Count == 3

    // 三套对应的 Context（上下文）
    FD3D12CommandContext* GraphicsContext;     // 图形操作
    FD3D12CommandContext* AsyncComputeContext; // 异步计算
    FD3D12CommandContext* CopyContext;         // 资源拷贝

    // 立即执行上下文（同步等待 GPU 结果）
    FD3D12CommandContext* ImmediateContext;
};
```

命令队列类型枚举：

```cpp
enum ED3D12QueueType
{
    Direct   = 0,  // 图形 + 计算（最常用）
    Copy     = 1,  // 资源拷贝
    Async    = 2,  // 异步计算
    Count    = 3
};
```

一个 GPU Device 对应三条队列，可以并行执行不同类型的工作。

---

### FD3D12Context：执行上下文

`FD3D12CommandContext` 继承自 `IRHICommandContext`，实现所有虚函数。核心成员：

```cpp
class FD3D12CommandContext : public IRHICommandContext
{
    FD3D12CommandListManager* CommandListManager; // CommandList 分配器
    FD3D12Payload*            CurrentPayload;     // 当前正在积累的 Payload
    FD3D12StateCache          StateCache;         // 状态缓存（最关键！）
    // ...
};
```

- `StateCache`：在真正调用 D3D12 API 前，所有状态设置都先写入 StateCache
- `CurrentPayload`：积累当前帧的所有 CommandList，到时候批量提交

---

### FD3D12StateCache：状态缓存与延迟提交

`FD3D12StateCache` 是 D3D12 后端最关键的优化设计：

**写入阶段**（RHI 线程调用 SetShaderXxx 等时）：

```cpp
void FD3D12CommandContext::RHISetShaderSampler(
    FRHIGraphicsShader* Shader,
    uint32 SamplerIndex,
    FRHISamplerState* NewState)
{
    // 只是写入 StateCache，不调用任何 D3D12 API
    StateCache.SetShaderSampler(ShaderFrequency, NewState, SamplerIndex);
}

void FD3D12CommandContext::RHISetViewport(float X, float Y, float W, float H, float MinZ, float MaxZ)
{
    // 只是拷贝到 StateCache 中的 GraphicsState.Viewport
    FMemory::Memcpy(&StateCache.GraphicsState.Viewport, ...);
}
```

**提交阶段**（D3D12 专属线程触发 `Apply()`）：

```cpp
void FD3D12StateCache::ApplyState(bool bForceState)
{
    // 根据 GraphicsState / ComputeState 中积累的信息
    // 1. 设置 RootSignature
    CommandList->SetGraphicsRootSignature(RootSignature);

    // 2. 设置 PSO（Pipeline State Object）
    CommandList->SetPipelineState(PipelineStateObject);

    // 3. 批量设置 ConstantBuffers、SRV、UAV、Samplers
    //    通过 D3D12 的 Root Descriptor Table

    // 4. 设置 Vertex Buffer
    CommandList->IASetVertexBuffers(0, NumSlots, VertexBufferViews);

    // 5. 设置 Viewport / ScissorRect
    CommandList->RSSetViewports(1, &Viewport);
    CommandList->RSSetScissorRects(1, &ScissorRect);

    // ... 其他状态全部在此一次性提交
}
```

**StateCache 内部关键状态结构**（简化）：

```cpp
struct FD3D12StateCache
{
    struct FGraphicsState
    {
        FD3D12GraphicsPipelineState* CurrentPipelineStateObject; // PSO
        FD3D12RootSignature*         CurrentRootSignature;       // Root Signature
        D3D12_VIEWPORT               Viewport;
        D3D12_RECT                   ScissorRect;
        FD3D12VertexBufferView       VertexBuffers[MAX_VERTEX_STREAMS];
        FD3D12ShaderResourceView*    SRVs[SF_NumFrequencies][MAX_SRVS];
        FD3D12UnorderedAccessView*   UAVs[MAX_UAVS];
        FD3D12SamplerState*          Samplers[SF_NumFrequencies][MAX_SAMPLERS];
        ID3D12Resource*              IndexBuffer;
        D3D12_PRIMITIVE_TOPOLOGY     PrimitiveTopology;
        FD3D12BlendState*            BlendState;
        FD3D12DepthStencilState*     DepthStencilState;
        FD3D12RasterizerState*       RasterizerState;
        // ... 大量状态
    } GraphicsState;

    struct FComputeState
    {
        // Compute Shader 相关状态
    } ComputeState;

    void ApplyState(bool bForce = false); // 真正写入 D3D12 CommandList
};
```

**两层缓存结构**：

```
RHI 线程调用 RHISetXxx()
        ↓
   写入 StateCache（只是内存写）
        ↓
   D3D12 线程执行 ApplyState()
        ↓
   真正调用 ID3D12GraphicsCommandList->SetXxx()
        ↓
   GPU 执行
```

### FD3D12Payload：一组工作的载体

```cpp
struct FD3D12Payload
{
    ED3D12QueueType      QueueType;                        // 目标队列类型
    TArray<FD3D12CommandList*> CommandLists;               // 一批 CommandList
    TArray<FD3D12SyncPoint>    WaitSyncPoints;             // 提交前需要等待的同步点
    // ...
};
```

`FD3D12Payload` 把一帧中属于同一个队列的所有 `ID3D12CommandList` 打包，交给 D3D12 线程统一提交（`ID3D12CommandQueue::ExecuteCommandLists()`）。

## D3D12 专属线程与执行流程

### 线程初始化

引擎启动时，D3D12 RHI 的初始化函数（大约是 `FD3D12DynamicRHI::Init()`）会创建一个专属的工作线程：

```cpp
// 初始化阶段（伪码）
FRunnableThread::Create(
    new FD3D12SubmissionThread(),  // 专门负责把 Payload 提交给 GPU
    TEXT("D3D12 Submission Thread")
);
```

该线程持续运行一个处理循环，不断检查各 Device 各 Queue 是否有待提交的 Payload。

---

### ProcessSomeMix：遍历执行

D3D12 线程的核心处理函数（命名类似 `ProcessSomeMixes` 或 `ProcessIncoming`，视 UE 版本而定）：

```cpp
void ProcessSomeMix()
{
    for (FD3D12Device* Device : AllDevices)
    {
        for (int32 QueueIndex = 0; QueueIndex < ED3D12QueueType::Count; QueueIndex++)
        {
            FD3D12Queue& Queue = Device->Queues[QueueIndex];

            // 取出当前等待提交的 Payload
            FD3D12Payload* Payload = Queue.DequeuePayload();
            if (!Payload) continue;

            // 执行 Payload 中的所有 CommandList
            ExecutePayload(Queue, Payload);
        }
    }
}

void ExecutePayload(FD3D12Queue& Queue, FD3D12Payload* Payload)
{
    // 应用状态缓存（实际写入 D3D12 CommandList）
    Payload->ApplyStateCache();

    // 收集 CommandList 数组
    TArray<ID3D12CommandList*> D3DCommandLists;
    for (FD3D12CommandList* CmdList : Payload->CommandLists)
    {
        // 关闭 CommandList 准备提交
        CmdList->Close();
        D3DCommandLists.Add(CmdList->GetD3DCommandList());
    }

    // 分批提交（防止单次提交数量超过 GPU 限制）
    constexpr uint32 MaxCommandListsPerSubmission = /* GPU 硬件上限 */;
    uint32 Offset = 0;
    while (Offset < D3DCommandLists.Num())
    {
        uint32 Count = FMath::Min(D3DCommandLists.Num() - Offset,
                                  MaxCommandListsPerSubmission);
        // 真正提交给 GPU 的 D3D12 API 调用
        Queue.GetD3DCommandQueue()->ExecuteCommandLists(
            Count,
            D3DCommandLists.GetData() + Offset
        );
        Offset += Count;
    }

    // CommandList 用完归还对象池
    for (FD3D12CommandList* CmdList : Payload->CommandLists)
    {
        Queue.GetCommandListPool().ReturnToPool(CmdList);
    }
}
```

**分批提交的原因**：D3D12 的 `ExecuteCommandLists()` 对单次提交的 CommandList 数量有上限，超出会导致 GPU 异常，因此需要分段循环提交。

---

### CommandList 对象池复用

`FD3D12CommandListPool`（或类似命名）实现了 CommandList 的池化：

```cpp
// 申请一个 CommandList（从池中取出或新建）
FD3D12CommandList* FD3D12CommandListPool::ObtainCommandList()
{
    if (AvailableCommandLists.IsEmpty())
    {
        // 池为空，新建一个
        return CreateNewCommandList();
    }
    return AvailableCommandLists.Pop();
}

// 归还 CommandList（清理后放回池中）
void FD3D12CommandListPool::ReturnToPool(FD3D12CommandList* CmdList)
{
    CmdList->Reset(); // 重置状态，准备下次使用
    AvailableCommandLists.Push(CmdList);
}
```

对象池避免了每帧反复 `new`/`delete` CommandList 的开销，D3D12 的 CommandList 创建成本较高。

## 整体调用链与时序图

### 命令从提交到执行的完整路径

```
游戏线程 (Game Thread)
    │
    │  ENQUEUE_RENDER_COMMAND([](FRHICommandListImmediate& RHICmdList) {
    │      RHICmdList.SetShaderParameter(...);
    │      RHICmdList.DrawIndexedPrimitive(...);
    │  })
    │
    ▼
渲染线程 (Render Thread) - RHI Thread
    │
    │  FRHICommandList::AllocateCommand<TRHILambdaCommand>()
    │      → *Tail = newCmd
    │      → Tail = &newCmd->Next        // 命令加入链表
    │
    │  [各种 RHICmdList.SetXxx()]
    │      → AllocateCommand<FRHICommandSetShaderParameter>()
    │      → ...（命令逐条入队）
    │
    │  FRHICommandList::Execute()
    │      → 遍历链表，调用每条命令的 ExecuteAndDestruct()
    │      → FRHICommandSetShaderParameter::Execute()
    │           → IRHICommandContext::RHISetShaderParameter()
    │                → FD3D12CommandContext::RHISetShaderParameter()
    │                     → StateCache.SetShaderParameter(...)  // 只写缓存！
    │
    ▼
D3D12 专属线程 (D3D12 Submission Thread)
    │
    │  loop: ProcessSomeMix()
    │      → 遍历 AllDevices
    │           → 遍历 3 条 Queue（Direct / Copy / Async）
    │                → DequeuePayload()
    │                → ExecutePayload()
    │                     → StateCache.ApplyState()       // 真正写 D3D12 CommandList
    │                     → CommandQueue->ExecuteCommandLists()  // 提交给 GPU 驱动
    │
    ▼
GPU 驱动 / GPU 硬件
    │
    └── 实际渲染执行
```

### 三层缓冲时序图

```
时间轴 ──────────────────────────────────────────────→

Frame N:
  Game Thread:    [逻辑更新] → ENQUEUE_RENDER_COMMAND(...)
                                       ↓
  Render Thread:                [Execute RHI Commands] → 写 StateCache
                                                               ↓
  D3D12 Thread:                                      [ApplyState + Submit]
                                                               ↓
  GPU:                                                              [渲染]

Frame N+1:
  Game Thread:    [逻辑更新] → ENQUEUE_RENDER_COMMAND(...)
                  ...（同上，三个线程流水线并行）
```

## 关键宏展开

### ENQUEUE_RENDER_COMMAND

引擎外部最常见的提交渲染命令宏，其本质展开如下：

```cpp
// 宏定义（简化版）
#define ENQUEUE_RENDER_COMMAND(Type)                                    \
    struct Type##Command                                                 \
    {                                                                    \
        template <typename LAMBDA>                                       \
        static FORCEINLINE void EnqueueLambda(LAMBDA&& Lambda)          \
        {                                                                \
            if (IsInRenderingThread())                                   \
            {                                                            \
                // 渲染线程：立即执行
                Lambda(GetImmediateCommandList_ForRenderCommand());     \
            }                                                            \
            else                                                         \
            {                                                            \
                // 游戏线程：加入渲染命令队列
                ALLOC_COMMAND_CL(GetCommandList(), TRHILambdaCommand<LAMBDA>)
                    (MoveTemp(Lambda));                                  \
            }                                                            \
        }                                                                \
    };                                                                   \
    Type##Command::EnqueueLambda

// 使用示例
ENQUEUE_RENDER_COMMAND(MyRenderCommand)(
    [MyParam](FRHICommandListImmediate& RHICmdList)
    {
        // 这里的代码运行在渲染线程
        RHICmdList.SetShaderParameter(...);
    }
);
```

### ALLOC_COMMAND / AllocCommand

```cpp
// 在 CommandList 上分配一条具体类型的命令（placement new + 自动入队）
#define ALLOC_COMMAND(...) \
    new (AllocCommand(sizeof(__VA_ARGS__), alignof(__VA_ARGS__))) __VA_ARGS__

// 典型调用形式（以 SetShaderParameter 为例）
void FRHICommandList::SetShaderParameter(...)
{
    // 1. 分配命令对象（已自动挂链表尾）
    auto* Cmd = ALLOC_COMMAND(FRHICommandSetShaderParameter);
    // 2. 初始化命令成员
    Cmd->Shader  = Shader;
    Cmd->Offset  = Offset;
    Cmd->NewValue = Value;
    // 注：此时未调用任何 D3D12 API
}
```

## 总结

本集揭示了 UE5 渲染命令体系的**四层设计**：

```
第一层：FRHICommandBase 链表
    └── 命令对象通过 Next 指针串联，AllocateCommand 自动入队

第二层：FRHICommandList / IRHICommandContext
    └── 提供平台无关的渲染接口
    └── Execute() 遍历链表，调用每条命令的 ExecuteAndDestruct()

第三层：FD3D12CommandContext + FD3D12StateCache
    └── RHI 接口的 D3D12 实现：所有状态写入 StateCache，不立即提交
    └── StateCache.ApplyState() 才是真正调用 D3D12 CommandList API 的地方

第四层：FD3D12Payload + D3D12 专属线程
    └── 积累一批 CommandList 打包成 Payload
    └── D3D12 线程循环：遍历 Device → Queue → Payload → ExecuteCommandLists()
    └── CommandList 通过对象池复用，避免频繁分配
```

**核心思想**：

| 阶段 | 操作 | 线程 |
|------|------|------|
| 命令创建 | `AllocateCommand()` 入链表 | 游戏线程 / 渲染线程 |
| RHI 执行 | `Execute()` 遍历链表，状态写 StateCache | RHI 渲染线程 |
| D3D12 提交 | `ApplyState()` + `ExecuteCommandLists()` | D3D12 专属线程 |
| GPU 执行 | 实际渲染 | GPU 硬件 |

后续系列将在此基础上继续分析上层如何把多条命令组装成渲染 Pass，以及多个 Pass 如何构成完整渲染流程。
